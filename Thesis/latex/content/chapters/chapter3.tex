\chapter{วิธีการศึกษา}
\label{chapter3}

\section{การออกแบบระบบ}
\label{sec:ch3_system_design}
\noindent
จากการศึกษาบริบทของสถานดูแลและปัญหาการขาดแคลนบุคลากรในการดูแลผู้ใช้เก้าอี้รถเข็นและผู้สูงอายุในบทที่ 1 ระบบต้นแบบ WheelSense จึงถูกออกแบบขึ้นเพื่อแก้ปัญหาด้านความปลอดภัยและการรับรู้สถานการณ์ โดยมีเป้าหมายเพื่อช่วยให้ผู้ดูแลสามารถติดตามตำแหน่งและรับรู้เหตุผิดปกติได้อย่างรวดเร็ว
สถาปัตยกรรมของระบบรองรับการทำงานแบบครบวงจร ตั้งแต่การติดตามตำแหน่งแบบเรียลไทม์ การแจ้งเตือนเหตุการณ์ ไปจนถึงการใช้ระบบผู้ช่วยอัจฉริยะ (AI) โครงสร้างสถาปัตยกรรมแบ่งออกเป็น 3 ระดับหลัก ได้แก่
\begin{enumerate}[nosep]
    \item \textbf{ระดับอุปกรณ์ภาคสนาม (Firmware Layer)} ประกอบด้วยเซ็นเซอร์บนรถเข็น อุปกรณ์สวมใส่ โทรศัพท์สมาร์ทโฟน และอุปกรณ์อ้างอิงตำแหน่ง ทำหน้าที่อ่านค่าพิกัดและข้อมูลสัญญาณชีพ
    \item \textbf{ระดับประมวลผลส่วนกลาง (Server Layer)} ประกอบด้วย MQTT Broker, API Server, Database และ AI Service ทำหน้าที่รับข้อมูล วิเคราะห์ และประมวลผล
    \item \textbf{ระดับส่วนติดต่อผู้ใช้งาน (Frontend Layer)} ประกอบด้วยแดชบอร์ดเว็บและแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ ทำหน้าที่แสดงผลและโต้ตอบกับผู้ใช้งานตามบทบาท
\end{enumerate}

ในระดับอุปกรณ์ภาคสนาม ไมโครคอนโทรลเลอร์ M5StickC Plus2 ติดตั้งบนเก้าอี้รถเข็นเพื่ออ่านค่าเซ็นเซอร์วัดแรงเฉื่อย (IMU) แบบ 6 แกน และส่งข้อมูล telemetry ผ่าน Wi-Fi โดยใช้โปรโตคอล MQTT อุปกรณ์สวมใส่ Polar Verity Sense ทำหน้าที่วัดสัญญาณชีพผ่านเซ็นเซอร์แสง (PPG) และส่งข้อมูลอัตราการเต้นของหัวใจไปยังสมาร์ทโฟนของผู้ใช้หรือผู้ดูแลผ่าน Bluetooth Low Energy (BLE) อุปกรณ์ Node Tsimcam ทำหน้าที่เป็นจุดส่งสัญญาณ BLE beacon สำหรับอ้างอิงตำแหน่งภายในอาคาร พร้อมถ่ายภาพประจำจุดตามการตั้งค่า โดยสมาร์ทโฟนยังทำหน้าที่เป็นสะพานเชื่อมต่อข้อมูลสัญญาณชีพจากอุปกรณ์สวมใส่เข้าสู่ระบบเครือข่าย

ระดับประมวลผลส่วนกลางประกอบด้วย MQTT Broker (Eclipse Mosquitto) สำหรับรับส่งข้อมูล telemetry จากอุปกรณ์ภาคสนามแบบ Publish/Subscribe Backend API ที่พัฒนาด้วย FastAPI ทำหน้าที่จัดการตรรกะธุรกิจ ตรวจสอบสิทธิ์การเข้าถึง และประมวลผลคำขอจากฝั่งผู้ใช้งาน ฐานข้อมูล PostgreSQL จัดเก็บทั้งข้อมูลที่มีโครงสร้างและข้อมูลกึ่งโครงสร้าง (JSONB) สำหรับ telemetry ข้อมูลการแจ้งเตือน และบันทึกตรวจสอบ (Audit Log) บริการ AI (Ollama ร่วมกับ FastMCP) ทำหน้าที่เป็นผู้ช่วยอัจฉริยะ รับคำถามจากผู้ใช้งาน วิเคราะห์บริบทระบบ และเรียกใช้เครื่องมือ (Tools) ในระบบผ่าน Model Context Protocol (MCP)

ระดับส่วนติดต่อผู้ใช้งานประกอบด้วยแดชบอร์ดเว็บที่พัฒนาด้วย Next.js และแอปพลิเคชันบนอุปกรณ์เคลื่อนที่ที่พัฒนาด้วย React Native ทั้งสองช่องทางรองรับการเข้าถึงตามบทบาท (Role-based) เช่น ผู้ดูแลระบบ พยาบาลหัวหน้า ผู้สังเกตการณ์ และผู้ใช้เก้าอี้รถเข็น โดยใช้ระบบตรวจสอบสิทธิ์ด้วย JWT ที่จัดเก็บในรูปแบบ HttpOnly cookie

การไหลของข้อมูลในระบบเริ่มจากอุปกรณ์ภาคสนามส่งข้อมูล telemetry ผ่านโปรโตคอล MQTT ไปยัง MQTT Broker จากนั้น Backend API ดึงข้อมูลไปประมวลผล จัดเก็บในฐานข้อมูล PostgreSQL และส่งต่อไปยังหน้าจอผู้ใช้งานผ่าน REST API หรือ WebSocket สำหรับการอัปเดตแบบเรียลไทม์ บริการ AI สื่อสารกับ Backend API เพื่อดึงบริบทระบบและสั่งการผ่านเครื่องมือที่กำหนดไว้ ขณะที่ Frontend ส่งคำสั่งและคำถามจากผู้ใช้งานไปยัง Backend API เพื่อประมวลผลต่อไป

\subsection{โครงข่ายข้อมูล}
\label{sec:ch3_sensor_network}

M5StickC Plus2 เชื่อมต่อกับเซิร์ฟเวอร์ส่วนกลางผ่านเครือข่าย Wi-Fi โดยใช้โปรโตคอล MQTT เพื่อส่งข้อมูล telemetry ความถี่สูง โปรโตคอล MQTT ถูกเลือกเนื่องจากมี overhead ต่ำ รองรับการเชื่อมต่อที่ไม่เสถียร และสามารถกำหนดหัวข้อ (Topic) เฉพาะสำหรับแต่ละอุปกรณ์ได้ เช่น \texttt{WheelSense/data/<device\_id>} นอกจากนี้ อุปกรณ์ยังสมัครรับหัวข้อคำสั่งควบคุม (\texttt{WheelSense/<device\_id>/control}) และหัวข้ออัปเดตการตั้งค่า (\texttt{WheelSense/config/...}) เพื่อรับคำสั่งจากเซิร์ฟเวอร์แบบเรียลไทม์

เมื่อเชื่อมต่อ Wi-Fi ไม่สำเร็จ ระบบจะพยายามเชื่อมต่อใหม่ด้วยระยะห่างที่เพิ่มขึ้นแบบเอกซ์โพเนนเชียล (exponential backoff) สูงสุด 60 วินาที สำหรับการเชื่อมต่อ MQTT หากหลุดการเชื่อมต่อจะพยายามเชื่อมต่อใหม่ทุก 2 วินาที และเพิ่มระยะห่างขึ้นจนถึง 60 วินาทีเช่นกัน การสมัครรับหัวข้อหลังเชื่อมต่อ MQTT สำเร็จประกอบด้วยหัวข้อการตั้งค่า คำสั่งควบคุม และการยืนยัน (acknowledgement)

\subsection{การประมวลผลข้อมูล}
\label{sec:ch3_sensor_processing}

ข้อมูลดิบจากเซ็นเซอร์ IMU อ่านผ่านไลบรารี M5.Imu โดยใช้ชิป MPU-6886 ซึ่งให้ค่าความเร่งและอัตราการหมุนแบบ 3 แกน ก่อนการใช้งานจริง ระบบจะคาริเบรตค่าออฟเซตของไจโรสโคป (Gyro Z Offset) โดยเก็บตัวอย่าง 1 วินาทีและหาค่าเฉลี่ย เพื่อลดค่าเบี่ยงเบี้ยนคงที่ (DC bias)

ในแต่ละรอบการอ่านค่า ระบบนำค่าไจโรสโคปแกน Z ผ่านตัวกรองแบบเอกซ์โพเนนเชียลมูฟวิงอเวอเรจ (EMA Low-pass Filter) เพื่อลดสัญญาณรบกวน จากนั้นใช้ Deadband ตัดค่าที่ต่ำกว่าเกณฑ์ออก แล้วแปลงค่าเป็นระยะทางและความเร็วโดยอินทิเกรตตามเวลา โดยอ้างอิงรัศมีล้อรถเข็นที่ตั้งค่าไว้ ความเร็วคำนวณผ่านหน้าต่างเลื่อน (sliding window) และมีกลไกลดค่าความเร็วลง (decay) เมื่อไม่พบการเคลื่อนที่เป็นเวลากำหนด เพื่อลด drift

นอกจาก IMU อุปกรณ์ยังสแกนสัญญาณ BLE beacon รอบๆ ตัวแบบ active scan โดยแยกชื่ออุปกรณ์ที่ขึ้นต้นด้วย \texttt{WSN\_} แล้วบันทึกค่า RSSI และ MAC address ไว้ในรายการ พร้อมกำจัดรายการที่ไม่พบนานเกินกำหนด

ข้อมูลทั้งหมดถูกจัดรูปแบบเป็น JSON ประกอบด้วย ข้อมูล IMU (ความเร่ง 6 แกน) ข้อมูล motion (ระยะทาง ความเร็ว ความเร่ง ทิศทาง) รายการ RSSI จาก beacon ที่พบ และสถานะแบตเตอรี่ จากนั้นส่งผ่าน MQTT ด้วยอัตราที่ปรับตามสถานะการทำงาน คือ 50 มิลลิวินาทีขณะบันทึกการเคลื่อนที่ หรือตามช่วงเวลาปกติเมื่ออยู่นิ่ง

\section{การจำแนกตำแหน่ง}
\label{sec:ch3_localization}

\subsection{ฮาร์ดแวร์}
\label{sec:ch3_localization_hardware}

ระบบจำแนกตำแหน่งภายในอาคารใช้อุปกรณ์ Node Tsimcam ซึ่งเป็นโหนดอเนกประสงค์ที่ทำหน้าที่ทั้งเป็น BLE beacon และกล้องถ่ายภาพในตัวเดียวกัน อุปกรณ์นี้ติดตั้งประจำจุดตามโซนหรือห้องต่างๆ บนเพดาน โดยส่งสัญญาณโฆษณา BLE อย่างต่อเนื่องพร้อมระบุชื่อโหนด (เช่น \texttt{WSN\_001}) เพื่อให้อุปกรณ์ภาคสนามสามารถรับค่าความเข้มสัญญาณ (RSSI) และนำไปใช้อ้างอิงตำแหน่ง

นอกจากการส่งสัญญาณ beacon Node Tsimcam ยังติดตั้งกล้องดิจิตอลที่สามารถถ่ายภาพนิ่งและส่งรูปภาพไปยังเซิร์ฟเวอร์ส่วนกลางผ่านโปรโตคอล MQTT โดยใช้วิธีแบ่งรูปภาพเป็นชิ้นเล็กๆ (chunked base64) หรือส่งแบบไบนารีโดยตรงผ่าน MQTT publish รูปภาพที่ได้ใช้สำหรับการตรวจสอบภาพในพื้นที่และยืนยันสถานะของผู้ใช้งานแบบ offline ร่วมกับข้อมูลตำแหน่งจากสัญญาณ RSSI

\subsection{การประมวลผลข้อมูล}
\label{sec:ch3_localization_processing}

กระบวนการประมวลผลตำแหน่งทำงานที่ฝั่งเซิร์ฟเวอร์ โดยรับค่า RSSI จากอุปกรณ์ภาคสนาม 2 ชนิด คือ M5StickC Plus2 ที่ติดตั้งบนเก้าอี้รถเข็น และสมาร์ทโฟนของผู้ใช้งานหรือผู้ดูแล ทั้งสองอุปกรณ์สแกนหา BLE beacon จาก Node Tsimcam ที่กระจายอยู่ทั่วอาคาร และส่งชุดค่า RSSI พร้อมรหัสโหนดและเวลาไปยังเซิร์ฟเวอร์ผ่าน MQTT

เซิร์ฟเวอร์จัดเก็บชุดค่า RSSI ดังกล่าวในฐานข้อมูล PostgreSQL รูปแบบ JSONB และใช้อัลกอริทึม K-Nearest Neighbors (K-NN) ในการประมวลผลตำแหน่ง โดยเปรียบเทียบชุดค่า RSSI ที่ได้รับในปัจจุบันกับชุดข้อมูลอ้างอิง (Radio Map) ที่บันทึกไว้ล่วงหน้าจากการสำรวจแต่ละโซน การคำนวณระยะห่างทางสถิติใช้ Euclidean Distance ดังสมการ~\ref{eq:ch3_euclidean} แล้วเลือกจุดอ้างอิงที่ใกล้เคียงที่สุดจำนวน $k$ จุด เพื่อโหวตน้ำหนักหาตำแหน่งหรือโซนที่ผู้ใช้แต่ละคนอยู่ในขณะนั้น ตามสมการ~\ref{eq:ch3_weighted_knn} ข้อมูลจากทั้ง M5StickC และสมาร์ทโฟนช่วยเพิ่มความถูกต้องเนื่องจากเซิร์ฟเวอร์สามารถรวมค่า RSSI จากหลายมุมมองและแก้ไขค่าที่ผิดปกติได้

\begin{equation}
d_j = \sqrt{\sum_{i=1}^{m} (x_i - r_{ji})^2}
\label{eq:ch3_euclidean}
\end{equation}

จากนั้นระบบจะใช้อัลกอริทึม K-Nearest Neighbors (K-NN) โดยเลือกจุดอ้างอิงที่มีระยะห่างทางสถิติน้อยที่สุดจำนวน $k$ จุด แล้วทำการโหวตน้ำหนักเพื่อหาพิกัดหรือโซนที่น่าจะเป็นไปได้มากที่สุด

\begin{equation}
\hat{y} = \arg\max_{y \in \mathcal{Y}} \sum_{i=1}^{k} \frac{1}{d_i} \cdot \mathbb{1}_{[y_i = y]}
\label{eq:ch3_weighted_knn}
\end{equation}

\subsection{สถาปัตยกรรมโดยรวม}
\label{sec:ch3_server_architecture}

สถาปัตยกรรมเซิร์ฟเวอร์ออกแบบบนแนวคิดไมโครเซอร์วิสผ่าน Docker Compose เพื่อให้ระบบสามารถขยายขนาดและดูแลรักษาได้ง่าย โดยประกอบด้วย 4 บริการหลัก ได้แก่ MQTT Broker (Eclipse Mosquitto) สำหรับจัดการการส่งข้อความแบบ Publish/Subscribe, Backend API (FastAPI) สำหรับจัดการตรรกะธุรกิจ, Database (PostgreSQL 16) สำหรับเก็บข้อมูล และ AI Service (Ollama + FastMCP) สำหรับประมวลผลผู้ช่วยอัจฉริยะ

ระบบเซิร์ฟเวอร์ทั้งหมดถูกนำไปใช้งานบน Raspberry Pi 5 ซึ่งเป็นคอมพิวเตอร์บอร์ดขนาดเล็ก (Single Board Computer) ที่มีหน่วยความจำ RAM 8 GB และใช้ชิป ARM Cortex-A76 Quad-core ความถี่ 2.4 GHz การเลือกใช้ Raspberry Pi 5 ช่วยลดต้นทุนด้านฮาร์ดแวร์และพลังงาน ในขณะเดียวกันก็ยังสามารถรันบริการต่างๆ ผ่าน Docker Compose ได้อย่างมีประสิทธิภาพ ทำให้เหมาะสำหรับการทำระบบเซิร์ฟเวอร์ส่วนกลางในโครงการ IoT ที่ต้องการความยืดหยุ่นและความสามารถในการปรับขนาด

การจัดการชีวิตวงจรของแอปพลิเคชัน (Application Lifecycle) ใช้ Lifespan Manager ของ FastAPI ซึ่งเริ่มต้นด้วยการตรวจสอบการตั้งค่าระบบ (Runtime Settings Validation) จากนั้นตรวจสอบการเชื่อมต่อฐานข้อมูล สร้างบัญชีผู้ดูแลระบบเริ่มต้นหากยังไม่มี เริ่มงาน MQTT Listener เป็นพื้นหลัง (Background Task) และเปิดใช้งาน Retention Scheduler ตามเงื่อนไข เมื่อปิดระบบจะยกเลิกงาน MQTT และหยุด Scheduler อย่างเป็นระเบียบ

ชั้นบริการ (Service Layer) แยกตรรกะธุรกิจออกจากชั้นข้อมูล โดยมีคลาส CRUDBase ทั่วไปที่บังคับใช้ขอบเขต Workspace (Workspace Isolation) สำหรับการสร้าง อ่าน แก้ไข และลบข้อมูล ชั้นนี้ประสานงานระหว่างตัวจัดการเส้นทาง (Router) กับชั้นความคงทนข้อมูล (Persistence Layer) เพื่อให้สามารถทดสอบและบำรุงรักษาโค้ดได้ง่ายขึ้น

\subsection{แอปพลิเคชันแบ็กเอนด์ FastAPI}
\label{sec:ch3_fastapi}

แอปพลิเคชันแบ็กเอนด์พัฒนาด้วย FastAPI ซึ่งเป็นเฟรมเวอร์กแบบอะซิงโครนัสรองรับคำขอพร้อมกันได้สูง โดยใช้ Python Async/Await พร้อมกับไลบรารี Pydantic สำหรับตรวจสอบความถูกต้องของข้อมูล (Data Validation) ตั้งแต่ขั้นเข้า ส่วนประกอบหลักประกอบด้วย

\begin{enumerate}
    \item \textbf{Lifespan Manager:} จัดการชีวิตวงจรการเริ่มต้นและปิดระบบ ตรวจสอบการตั้งค่า เริ่ม MQTT Listener และ Retention Scheduler
    \item \textbf{Dependency Injection:} ใช้ FastAPI Depends สำหรับฉีดบริการที่ใช้ร่วมกัน เช่น ฐานข้อมูล ตัวตรวจสอบสิทธิ์ และ MQTT Client ทำให้สามารถทดสอบหน่วยย่อย (Unit Testing) ได้โดยง่าย
    \item \textbf{Router Organization:} แบ่งเส้นทาง API ตามโดเมนข้อมูล เช่น authentication, devices, rooms, users, workspaces, alerts และ AI ช่วยให้โครงสร้างชัดเจนและขยายได้
    \item \textbf{Error Handling:} ใช้รูปแบบคำตอบข้อผิดพลาดมาตรฐาน (Standardized Error Envelope) รหัส HTTP ที่เหมาะสม และข้อความภาษาไทยสำหรับผู้ใช้
\end{enumerate}

\subsection{ช่องทางสื่อสารและการรับข้อมูล}
\label{sec:ch3_server_channels}

การรับส่งข้อมูลในระบบเซิร์ฟเวอร์อาศัยช่องทางสื่อสาร 3 รูปแบบหลัก ดังนี้

\textbf{MQTT (Message Queuing Telemetry Transport)} ใช้สำหรับการรับส่งข้อมูล telemetry ความถี่สูงจากอุปกรณ์ภาคสนามแบบ Publish/Subscribe เซิร์ฟเวอร์ใช้ไลบรารี aiomqtt ซึ่งรองรับการทำงานแบบอะซิงโครนัส เพื่อรับข้อมูลจากอุปกรณ์หลายเครื่องพร้อมกันโดยไม่บล็อกเธรด หัวข้อหลักที่ใช้ประกอบด้วย \texttt{WheelSense/data} สำหรับข้อมูลเซ็นเซอร์ \texttt{WheelSense/camera/+/registration} สำหรับลงทะเบียนภาพถ่าย และ \texttt{WheelSense/+/ack} สำหรับยืนยันการได้รับคำสั่ง

\textbf{REST API} ใช้สำหรับการเรียกดูข้อมูลผู้ใช้ ประวัติ และคำสั่งจาก Dashboard แบบ Request/Response โดยรองรับการยืนยันตัวตนด้วย OAuth2 และการควบคุมสิทธิ์ตามบทบาท (RBAC)

\textbf{WebSocket} ใช้สำหรับการรับข้อมูลอัปเดตแบบเรียลไทม์ เช่น การแจ้งเตือนเหตุฉุกเฉิน และสตรีมมิงคำตอบจาก AI แบบ Full-duplex

ระบบรับข้อมูล telemetry ทำงานตามลำดับขั้น ได้แก่ รับแพ็กเก็ตจาก MQTT Broker ตรวจสอบรูปแบบ JSON ตรวจสอบความถูกต้องของข้อมูลตามแบบจำลอง (Pydantic Model) ประมวลผลตรรกะธุรกิจ เช่น ตรวจจับการล้ม และบันทึกลงฐานข้อมูล นอกจากนี้ยังมีระบบรวมรูปภาพจากกล้องแบบแยกชิ้น (Camera Photo Chunking) ซึ่งรวบรวมชิ้นส่วนภาพแล้วประกอบเป็นภาพสมบูรณ์ก่อนจัดเก็บ

\subsection{ฐานข้อมูล PostgreSQL}
\label{sec:ch3_server_database}

ฐานข้อมูลระบบใช้ PostgreSQL 16 ร่วมกับ SQLAlchemy 2.0 แบบอะซิงโครนัส (Async SQLAlchemy) ผ่านไดรเวอร์ asyncpg เพื่อรองรับการทำงานพร้อมกันของคำขอจำนวนมากโดยไม่บล็อกเธรด การเชื่อมต่อใช้ Connection Pooling ที่มีการตั้งค่า pool\_size และ max\_overflow เหมาะสมกับภาระงานของระบบ

การจัดการเวอร์ชันของโครงสร้างฐานข้อมูลใช้ Alembic ซึ่งเป็นชั้นการย้ายข้อมูล (Migration Layer) แบบ incremental ทำให้สามารถปรับเปลี่ยนโครงสร้างตารางในเวอร์ชันต่อๆ ไปได้อย่างมีระเบียบ แต่ละเวอร์ชันมีการระบุ dependencies และ revision ID เพื่อป้องกันการทำงานซ้ำซ้อน

ชนิดข้อมูล JSONB ถูกใช้เป็นหลักสำหรับเก็บข้อมูลกึ่งโครงสร้าง เช่น payload ของ telemetry ข้อมูลการแจ้งเตือนแบบยืดหยุ่น และประวัติ Audit Log ทำให้สามารถรองรับรูปแบบข้อมูลจากเซ็นเซอร์ที่อาจปรับเปลี่ยนในอนาคตโดยไม่ต้องแก้ไขโครงสร้างตาราง

เอนทิตีหลักในฐานข้อมูลประกอบด้วย Workspace (แยกขอบเขตองค์กร), User (ผู้ใช้และบทบาท), AuthSession (เซสชันการเข้าใช้งาน), Device (อุปกรณ์ IoT), Room (ห้องและสถานที่), และ Alert (เหตุการณ์แจ้งเตือน) ทุกตารางที่อยู่ภายใต้ Workspace มีคอลัมน์ workspace\_id เพื่อบังคับใช้ขอบเขตข้อมูล (Workspace Isolation) ป้องกันไม่ให้ผู้ใช้จากองค์กรหนึ่งเข้าถึงข้อมูลขององค์กรอื่น

\subsection{ระบบควบคุมอุปกรณ์ Home Assistant}
\label{sec:ch3_homeassistant}

Home Assistant เป็นแพลตฟอร์มโอเพนซอร์สสำหรับการควบคุมอุปกรณ์ IoT และอุปกรณ์อัจฉริยะภายในบ้าน ในระบบ WheelSense Home Assistant ทำหน้าที่เป็นเกตเวย์ควบคุมอุปกรณ์อิเล็กทรอนิกส์ภายในห้อง เช่น หลองไฟ ปลั๊กอัจฉริยะ และอุปกรณ์อื่นๆ ผ่านการเชื่อมต่อผ่าน MQTT หรือ REST API

การเชื่อมต่อระหว่าง WheelSense และ Home Assistant ทำผ่าน MCP Protocol โดย FastAPI ส่งคำสั่งไปยัง Home Assistant เพื่อควบคุมอุปกรณ์ ตัวอย่างเช่น เมื่อผู้ใช้สั่ง "เปิดไฟในห้อง 101" ผ่าน AI Assistant ระบบจะส่งคำสั่งไปยัง Home Assistant เพื่อเปิดหลอดไฟที่กำหนดไว้ในห้องดังกล่าว

Home Assistant รองรับอุปกรณ์หลากหลายประเภทผ่านการเชื่อมต่อที่หลากหลาย เช่น Zigbee, Z-Wave, WiFi และ Bluetooth ทำให้สามารถขยายระบบได้อย่างยืดหยุ่น ในระบบ WheelSense ใช้ Home Assistant เพื่อจัดการอุปกรณ์ที่ไม่ได้เชื่อมต่อโดยตรงกับระบบหลัก หรืออุปกรณ์ที่ต้องการการควบคุมเฉพาะทาง

\subsection{ระบบรับรองความถูกต้องและการควบคุมสิทธิ์}
\label{sec:ch3_server_security}

ระบบความปลอดภัยออกแบบเป็นเลเยอร์ (Layered Security) ซึ่งครอบคลุมตั้งแต่การยืนยันตัวตนผู้ใช้ไปจนถึงการควบคุมสิทธิ์การเข้าถึงข้อมูล

\textbf{การยืนยันตัวตนด้วย JWT} ระบบใช้ JSON Web Token (JWT) สำหรับยืนยันตัวตนแบบ Stateless โดยผู้ใช้ลงชื่อเข้าใช้งานด้วยชื่อผู้ใช้และรหัสผ่าน รหัสผ่านถูกเก็บด้วยการแฮชแบบ bcrypt พร้อมค่า Salt รอบการทำงานที่ปลอดภัย จากนั้นระบบออกโทเคนเข้าถึง (Access Token) และโทเคนรีเฟรช (Refresh Token) ให้กับผู้ใช้

\textbf{การจัดการเซสชัน} ระบบเก็บเซสชันการเข้าใช้งานในฐานข้อมูล (AuthSession) พร้อมข้อมูลรุ่นเบราว์เซอร์ หมายเลข IP และสถานะ Active/Revoked ผู้ใช้สามารถยกเลิกเซสชันเฉพาะหรือทั้งหมดได้ ทำให้สามารถควบคุมการเข้าถึงของอุปกรณ์ที่เคยลงชื่อเข้าใช้ได้

\textbf{การควบคุมสิทธิ์ตามบทบาท (RBAC)} ระบบกำหนดบทบาทหลัก 3 ระดับ ได้แก่ System Admin (ดูแลระบบ), Workspace Admin (ผู้ดูแลองค์กร), และ User (ผู้ใช้ทั่วไป) สิทธิ์การเข้าถึง API ถูกกำหนดตามบทบาทในรูปแบบของยูนิตความสัมพันธ์ (Tuple) ระหว่างบทบาทกับสิทธิ์ เช่น (Admin, ดูแลผู้ใช้)

\textbf{ขอบเขตข้อมูล (Workspace Scoping)} ทุกคำขอที่ต้องการข้อมูลในส่วนแบ็กเอนด์ต้องระบุ Workspace ID ผ่านค่าคงที่ในเส้นทาง (Path) หรือค่าพารามิเตอร์ (Query Parameter) ระบบ Middleware ตรวจสอบว่าผู้ใช้มีสิทธิ์ในขอบเขตนั้นหรือไม่ก่อนประมวลผลคำขอ

\textbf{การป้องกันการเข้าถึง AI Service} บริการ MCP และ AI ใช้โทเคนแยกต่างหากที่มีขอบเขตเฉพาะ (Scope Narrowing) เพื่อป้องกันไม่ให้ AI Agent มีสิทธิ์เกินความจำเป็น ตัวอย่างเช่น โทเคนที่ใช้สำหรับอ่านข้อมูลอุปกรณ์จะไม่สามารถแก้ไขหรือลบข้อมูลได้

\section{ระบบปัญญาประดิษฐ์ (AI)}

\subsection{สถาปัตยกรรม}
\label{sec:ch3_ai_architecture}

สถาปัตยกรรมของ AI ทำงานเป็นระบบผู้ช่วยอัจฉริยะผ่านโครงสร้าง Pipeline 5 ชั้น (EaseAI) เพื่อลดข้อผิดพลาดในการประมวลผลและป้องกันการดำเนินการนอกขอบเขต โดยเชื่อมโยงกับโมเดลภาษาขนาดใหญ่ (LLM) ในเครื่อง (On-premise) เช่น โมเดล Gemma 4 ผ่านแพลตฟอร์ม Ollama นอกจากนี้ยังใช้ Model Context Protocol (MCP) เป็นส่วนเชื่อมต่อเพื่อให้ AI สามารถเข้าถึงข้อมูลในระบบ และกระทำต่อระบบผ่านเครื่องมือ (Tools) ได้อย่างปลอดภัย

\textbf{ตัวอย่างการทำงานของ Pipeline:} เมื่อผู้ใช้ส่งคำสั่ง ``"เปิดไฟในห้อง 101"`` เข้าสู่ระบบ กระบวนการจะดำเนินการดังนี้:
\begin{itemize}
    \item \textbf{Layer 1 (Intent Router):} Sentence-BERT จำแนกว่าเป็นคำสั่งควบคุมอุปกรณ์ (device\_control) พร้อมดึงพารามิเตอร์ ``room\_id=101`` และ ``action=turn\_on``
    \item \textbf{Layer 2 (Context Engine):} ระบบดึงข้อมูลจากฐานข้อมูลว่าห้อง 101 มี Smart Device ประเภท ``light`` พร้อม ``ha\_entity\_id=light.bedroom\_101`` และตรวจสอบสิทธิ์ของผู้ใช้
    \item \textbf{Layer 3 (Behavioral State):} ตรวจสอบประวัติการสนทนาและนโยบายความปลอดภัย ยืนยันว่าผู้ใช้มีสิทธิ์ควบคุมอุปกรณ์ในห้องดังกล่าว
    \item \textbf{Layer 4 (LLM Synthesis):} Gemma 4 สังเคราะห์แผนการดำเนินการ (execution plan) ตาม Schema ที่กำหนด ได้ผลลัพธ์เป็น JSON structure พร้อม tool call: ``control\_room\_smart\_device(room\_id=101, ha\_entity\_id=light.bedroom\_101, state=on)``
    \item \textbf{Layer 5 (Safety Check \& Execution):} ระบบตรวจสอบว่า tool call ปลอดภัย ไม่ขัดต่อนโยบาย จากนั้นส่งคำสั่งไปยัง Home Assistant ผ่าน MCP Protocol และบันทึกผลลัพธ์กลับใน ``pipeline\_events``
\end{itemize}

\subsection{Prompt}
\label{sec:ch3_ai_prompt}

การทำงานของ AI อาศัยโครงสร้าง Prompt ที่ระบุบทบาท ขอบเขตความรับผิดชอบ และนโยบายความปลอดภัยไว้อย่างชัดเจน โดย Prompt เริ่มต้น (System Prompt) จะกำหนดให้ AI สรุปข้อมูลหรืออธิบายเหตุการณ์ตามบริบทที่ดึงมาจากฐานข้อมูลและบริบทของผู้ใช้ (Context Engine) และปฏิบัติตามหลักการ Propose-Confirm-Execute เสมอ หากคำสั่งนั้นส่งผลให้เกิดการเปลี่ยนแปลงต่อระบบ

\textbf{System Prompt ตามบทบาทผู้ใช้:}

\begin{itemize}
    \item \textbf{admin:} ``You are EaseAI for WheelSense, assisting an IT administrator. Handle both general questions and WheelSense operations. For WheelSense-specific facts, rely on grounded data when available and never invent patient data.''
    \item \textbf{head\_nurse:} ``You are EaseAI for a head nurse. Handle both general questions and ward operations. Prioritize patient safety and concise action.''
    \item \textbf{supervisor:} ``You are EaseAI for a medical supervisor. Handle both general questions and clinical operations. Do not diagnose.''
    \item \textbf{observer:} ``You are EaseAI for floor staff. Handle both general questions and assigned-zone operations.''
    \item \textbf{patient:} ``You are EaseAI for a patient user. Handle both general questions and patient-facing assistance. Use simple language and encourage contacting staff for emergencies. When WheelSense tool results are provided for this session, you may use them to answer about this signed-in patient's own profile (including name and room assignment), vitals, timeline, assigned care team, messages, and alerts. Never infer or disclose other patients' data.''
\end{itemize}

\textbf{General Assistant Suffix (เพิ่มเติมสำหรับทุกบทบาท):}

``You can answer general questions, small talk, and broad knowledge queries as a normal assistant. When a request needs WheelSense live data or actions, use only grounded system data and do not invent facts. Do not attempt to run shell commands, modify application source code, or touch the host filesystem outside the published WheelSense MCP tools.''

ระบบมี Prompt หลัก 6 ประเภทตามบทบาทผู้ใช้:

\begin{enumerate}
    \item \textbf{admin-operations} - Prompt สำหรับการจัดการโครงสร้างพื้นฐานและพื้นที่ทำงาน เช่น จัดการผู้ใช้ อุปกรณ์ และการตั้งค่าระบบ
    \item \textbf{clinical-triage} - Prompt สำหรับการประเมินสภาพผู้พักอาศัยและการแจ้งเตือน เช่น สรุปสุขภาพล่าสุด และการตรวจสอบเหตุการณ์ผิดปกติ
    \item \textbf{observer-shift-assistant} - Prompt สำหรับเจ้าหน้าที่ปฏิบัติงานในพื้นที่ เช่น การตรวจสอบตำแหน่งผู้พักและการตอบสนองเหตุการณ์
    \item \textbf{patient-support} - Prompt สำหรับผู้พักอาศัย เช่น การสอบถามสุขภาพ การติดต่อเจ้าหน้าที่ และการควบคุมอุปกรณ์ในห้อง
    \item \textbf{device-control} - Prompt สำหรับการควบคุมอุปกรณ์และอุปกรณ์ในห้อง เช่น การสั่งงานกล้องและอุปกรณ์ IoT
    \item \textbf{facility-ops} - Prompt สำหรับการจัดการสิ่งอำนวยความสะดวกและผังอาคาร เช่น การดูผังชั้นและการจัดการห้อง
\end{enumerate}

\textbf{ตัวอย่าง Prompt ในระบบ:}

\begin{itemize}
    \item \textbf{คำถามทั่วไป:} ``ใครอยู่ในห้องโถงขณะนี้'' - AI จะใช้เครื่องมือ \texttt{list\_visible\_patients} และกรองตามตำแหน่งเพื่อตอบรายชื่อผู้พัก
    \item \textbf{คำถามเชิงสุขภาพ:} ``สุขภาวะของคุณสมชายเป็นอย่างไรบ้าง'' - AI จะใช้เครื่องมือ \texttt{get\_patient\_vitals} และ \texttt{get\_patient\_timeline} เพื่อดึงข้อมูลอัตราการเต้นหัวใจ ออกซิเจนในเลือด และประวัติการเคลื่อนไหว
    \item \textbf{คำสั่งการควบคุม:} ``เปิดไฟในห้อง 101'' - AI จะเสนอแผนการใช้เครื่องมือ \texttt{control\_room\_smart\_device} และรอการยืนยันจากผู้ใช้ก่อนดำเนินการ
    \item \textbf{คำสั่งการแจ้งเตือน:} ``ยืนยันการแจ้งเตือนการหกล้มของคุณสมชาย'' - AI จะใช้เครื่องมือ \texttt{acknowledge\_alert} เพื่ออัปเดตสถานะการแจ้งเตือน
    \item \textbf{คำถามเชิงระบบ:} ``สถานะระบบปกติหรือไม่'' - AI จะใช้เครื่องมือ \texttt{get\_system\_health} เพื่อตรวจสอบสถานะบริการทั้งหมด
\end{itemize}

\subsection{การเชื่อมต่อแบ็กเอนด์และประสิทธิภาพ}
\label{sec:ch3_hmi_api}

การเชื่อมต่อกับเซิร์ฟเวอร์ใช้ API Client แบบพิมพ์เขียม (Strongly Typed) ซึ่งห่อหุ้มคำขอ HTTP ด้วย JWT handling การหมดเวลา (Timeout) และการจัดการข้อผิดพลาดแบบมาตรฐาน คำขอทั้งหมดถูกนำผ่านเส้นทาง \texttt{/api} ของ Next.js เพื่อส่งต่อไปยัง FastAPI Backend โดยอัตโนมัติ

การจัดการข้อผิดพลาดรวมถึงการเปลี่ยนเส้นทางไปยังหน้าเข้าสู่ระบบเมื่อพบรหัส 401 และแสดงข้อความข้อผิดพลาดที่มีโครงสร้างจากเซิร์ฟเวอร์ นอกจากนี้ API Client ยังรองรับการตรวจสอบสิทธิ์แบบจำลองบทบาท (Impersonation) โดยส่งหัวข้อพิเศษพร้อมคำขอเพื่อให้ผู้ดูแลระบบสามารถตรวจสอบมุมมองของผู้ใช้รายอื่นได้

ด้านประสิทธิภาพ ระบบเปิดใช้งาน React Compiler ในการตั้งค่า Next.js เพื่อเพิ่มประสิทธิภาพการเรนเดอร์ React Query จัดการแคชข้อมูลด้วยนโยบายเวลาหมดอายุสั้น (Short Stale Time) และรีเฟรชอัตโนมัติเมื่อมีการเชื่อมต่อกลับ ช่วยลดการเรียกข้อมูลซ้ำจากเซิร์ฟเวอร์ ส่วนคอมโพเนนต์ที่มีขนาดใหญ่จะถูกโหลดแบบ Lazy Load เมื่อจำเป็น และรูปภาพใช้ Next/image เพื่อปรับขนาดและบีบอัดอัตโนมัติ

\subsection{ส่วน UI แอปพลิเคชันบนอุปกรณ์เคลื่อนที่}
\label{sec:ch3_hmi_mobile}

แอปพลิเคชันบนอุปกรณ์เคลื่อนที่ถูกออกแบบมาเพื่อบุคลากรที่ต้องเดินตรวจตราในพื้นที่ โดยมีหน้าจอที่เน้นความสะดวกในการอ่านและตอบสนองขณะเคลื่อนไหว (Mobile First) การออกแบบเน้นปุ่มขนาดใหญ่ที่สามารถใช้งานด้วยมือเดียว (One-handed Operation) และสีที่มีความต่างชัดเจนเพื่อให้อ่านได้ในสภาพแสงน้อย

หน้าจอหลักแบ่งเป็น 3 โซนหลัก ได้แก่ แถบสถานะด้านบนแสดงจำนวนการแจ้งเตือนที่ยังไม่รับทราบ บริเวณกลางแสดงรายการเหตุการณ์ล่าสุดพร้อมระดับความรุนแรง และแถบนำทางด้านล่างสำหรับสลับระหว่างมุมมองรายการเหตุการณ์ แผนที่ตำแหน่ง และการตั้งค่า

การแจ้งเตือนฉุกเฉิน (Push Notification) ส่งตรงถึงอุปกรณ์เคลื่อนที่แม้อยู่นอกแอปพลิเคชัน เมื่อผู้ใช้เปิดการแจ้งเตือน ระบบจะแสดงรายละเอียดผู้ป่วย ตำแหน่งที่เกิดเหตุ และเวลา พร้อมปุ่มรับทราบ (Acknowledge) การกดรับทราบจะอัปเดตสถานะในระบบแบ็กเอนด์ทันทีผ่าน REST API
