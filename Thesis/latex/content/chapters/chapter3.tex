\chapter{วิธีการวิจัย}
\label{chapter3}

บทนี้อธิบายการออกแบบระบบต้นแบบ WheelSense ในระดับที่ทำซ้ำได้ (reproducible engineering method)
ครอบคลุมการกำหนดเครื่องมือและอุปกรณ์ การออกแบบสถาปัตยกรรมและ data pipeline การกำหนดสัญญาแลกเปลี่ยนข้อมูล (data contract) การตั้งค่าพารามิเตอร์หลัก และโปรโตคอลการติดตั้ง เพื่อให้ระบบพร้อมใช้งานก่อนดำเนินการประเมิน
เนื้อหาในบทนี้ตอบคำถามสองประการ คือ (1) ระบบถูกสร้างและประกอบขึ้นอย่างไร (2) การประเมินผลเชิงวิจัยวัดเรื่องใดบ้าง
มีการแยกชัดระหว่าง (ก) การออกแบบและการทดสอบการทำงานเบื้องต้นระหว่างพัฒนา กับ (ข) \textbf{การประเมินที่ใช้เป็นหลักฐานในเล่ม}
การประเมินหลักในเล่มประกอบด้วย (1) \textbf{สี่ประเด็นด้าน UX/UI และ LLM/MCP} ตามหัวข้อ~\ref{sec:ch3_test_plan} และ (2) \textbf{การทดสอบเสริมด้านความทนทานของโมดูลระบุตำแหน่ง} โดยจำลองความเปลี่ยนแปลงของ RSSI (Gaussian noise injection) เพื่อเปรียบเทียบ KNN กับ XGBoost ภายใต้ข้อมูลและบริบทเดียวกัน
บทนี้เชื่อมโยงกับวัตถุประสงค์และขอบเขตในบทที่~\ref{chapter1} และอาศัยหลักการกับวรรณกรรมจากบทที่~\ref{chapter2} โดยไม่ทวนซ้ำนิยามหรือสมการของ IPS, RSSI fingerprinting และ KNN \cite{BLE_Fingerprint_Survey_IEEE,WiFi_BLE_IPS_Systematic_Review}
ผลเชิงตัวเลข ตัวชี้วัด และการอภิปรายข้อค้นพบ นำเสนอในบทที่~\ref{chapter4}

ลำดับการอ่านที่แนะนำ: (1) คำศัพท์และตาราง traceability (2) \textbf{สมมุติฐาน ตัวแปร และกรอบการประเมิน} ในหัวข้อ~\ref{sec:ch3_hypotheses} (3) เครื่องมือและสแต็กซอฟต์แวร์ (4) ภาพรวมสถาปัตยกรรมและสัญญา MQTT (5) บริบททดลองและข้อจำกัด (6) โมดูลอุปกรณ์ การระบุตำแหน่ง และเซิร์ฟเวอร์ (7) พารามิเตอร์และ \textbf{Profile-A} ในหัวข้อ~\ref{sec:ch3_eval_profile} (8) โปรโตคอลการติดตั้งและความพร้อมของระบบ (9) \textbf{แผนการประเมินผล} ในหัวข้อ~\ref{sec:ch3_test_plan}
หมายเหตุ: ตาราง failure modes อยู่ก่อนหัวข้อพารามิเตอร์ เพื่อสรุปความเสี่ยงของแต่ละระบบย่อยก่อนรวมเป็นตัวเลข Profile-A

\section{สมมุติฐาน ตัวแปร และกรอบการประเมิน}
\label{sec:ch3_hypotheses}

บทนี้เป็นการวิจัยเชิงวิศวกรรม (engineering research) ที่นำเสนอต้นแบบระบบสมบูรณ์และประเมินภายใต้ข้อกำหนดที่ทำซ้ำได้ ดังนั้นข้อความ ``สมมุติฐาน'' ในที่นี้หมายถึงข้อความที่สามารถตรวจสอบได้ด้วยตัวชี้วัดและโปรโตคอลที่ระบุ ไม่ใช่สมมุติฐานทางสถิติของประชากรทั่วไปในลักษณะสำรวจเชิงปริมาณขนาดใหญ่

\subsection{เส้นทางข้อมูลหลัก}
\label{sec:ch3_dataflow}

ลำดับเส้นทางหลักและจุดอ้างอิงเชิงปฏิบัติการในบทนี้มีดังนี้
\begin{enumerate}
    \item \textbf{อุปกรณ์บนเก้าอี้รถเข็น} — อ่าน IMU สร้าง motion feature สแกน BLE/RSSI และรวมสัญญาณสรีระศาสตร์ (HR) จาก Polar จากนั้นส่ง MQTT telemetry ที่มี \texttt{device\_id} และ timestamp (หัวข้อ~\ref{sec:ch3_wheelchair_module})
    \item \textbf{โหนดกล้อง} — ส่งภาพนิ่งแบบ chunked payload พร้อม metadata และรับคำสั่งควบคุมการถ่ายภาพ/พารามิเตอร์การส่ง (หัวข้อ~\ref{sec:ch3_camera_node})
    \item \textbf{MQTT broker} — เป็นช่องทางกลางรับข้อความจากอุปกรณ์และกระจายต่อให้บริการ ingestion/processing (หัวข้อ~\ref{sec:ch3_mqtt_contract})
    \item \textbf{บริการ ingestion + ฐานข้อมูล} — ตรวจสอบรูปแบบข้อมูล บันทึก time-series/event และจัดการความเป็นระเบียบของเวลา (หัวข้อ~\ref{sec:ch3_server_db})
    \item \textbf{บริการ localization} — ฝึก/ทำนายโซนหรือห้องด้วย RSSI fingerprinting + KNN และเก็บผลลัพธ์ (หัวข้อ~\ref{sec:ch3_indoor_loc})
    \item \textbf{แอปพลิเคชันผู้ใช้งาน} — web dashboard และ mobile app ดึงข้อมูลจาก API และรับอัปเดตแบบใกล้เวลาจริงด้วยกลไกที่กำหนด (หัวข้อ~\ref{sec:ch3_webapp},~\ref{sec:ch3_mobile})
    \item \textbf{Home Assistant + MCP/LLM} — ใช้ข้อมูลสถานะ/เหตุการณ์เพื่อ automation และการสรุป โดยจำกัดขอบเขตเครื่องมือและบันทึก audit log (หัวข้อ~\ref{sec:ch3_ha},~\ref{sec:ch3_llm_mcp})
\end{enumerate}

\section{บริบทการทดลองและข้อจำกัดการดำเนินงาน}
\label{sec:ch3_context_constraints}

\subsection{พื้นที่ทดสอบและผู้มีส่วนได้ส่วนเสีย}

การทดลองใช้พื้นที่ภายในอาคารที่อนุญาตให้ติดตั้ง BLE beacon และโหนด ESP32-S3 พร้อมกล้องได้อย่างปลอดภัย โดยมีผู้ใช้เก้าอี้รถเข็นและผู้ดูแลเป็นผู้มีส่วนได้ส่วนหลักในการประเมินความเหมาะสมของการแสดงผลและการแจ้งเตือน
ข้อกำหนดด้านความปลอดภัยของสถานที่และนโยบายการเก็บข้อมูลส่วนบุคคลต้องถูกนำมาพิจารณาร่วมกับการออกแบบระบบ

\subsection{ภาวะเดิมและข้อจำกัดเชิงเทคนิค}

ก่อนการบูรณาการแบบ end-to-end การทดลองต้นแบบก่อนหน้าพบข้อจำกัดที่ส่งผลต่อการออกแบบรุ่นปัจจุบัน ได้แก่ ความผันผวนของ RSSI ตามสภาพแวดล้อมและเวลา ความไม่สอดคล้องของ timestamp ระหว่างอุปกรณ์บนเก้าอี้ โหนดภาพ และเซิร์ฟเวอร์ และความซับซ้อนของการติดตั้งหลายบริการบนเซิร์ฟเวอร์เครื่องเดียว
ข้อจำกัดเหล่านี้ถูกจัดการในระดับสถาปัตยกรรมและพารามิเตอร์ตามที่อธิบายในหัวข้อ~\ref{sec:ch3_params} และผลจากการทดสอบจริงสรุปในบทที่~\ref{chapter4}

\subsection{ความเที่ยงตรงของเครื่องมือ ความเชื่อมั่นของการวัด และภัยคุกคามต่อความน่าเชื่อถือ}
\label{sec:ch3_validity_reliability}

\textbf{ความเที่ยงตรงเชิงโครงสร้าง (construct validity):} การประเมินผลเชิงวิจัยในเล่มนี้แบ่งเป็นกลุ่มหลักสองกลุ่ม
กลุ่มแรกคือ \textbf{สี่ประเด็นด้าน UX/UI และ LLM/MCP} ในหัวข้อ~\ref{sec:ch3_test_plan} ประกอบด้วย (1) \textbf{การรับรู้ต่อ UX/UI} ของผู้ดูแลจากแบบสอบถามและงานที่กำหนดบนระบบจริง (2) \textbf{ความสอดคล้องของการเลือกเครื่องมือ MCP} (IRR) (3) \textbf{ความใกล้เคียงเชิงข้อความ} ของเอาต์พุต LLM เทียบข้อความอ้างอิง (4) \textbf{เวลาแฝง} ของเส้นทาง LLM/MCP
กลุ่มที่สองคือ \textbf{การทดสอบเสริมด้านความทนทานของการระบุตำแหน่ง} ภายใต้การเปลี่ยนแปลงของ RSSI แบบจำลอง เพื่อสะท้อนความผันผวนของสัญญาณในสภาพแวดล้อมจริง
โมดูลระบุตำแหน่ง (KNN/max-RSSI) และ pipeline อื่น ๆ เป็นส่วนของการออกแบบต้นแบบเพื่อสนับสนุนบริบทบนแดชบอร์ดและงานของผู้ดูแล
การทดสอบเสริมกลุ่มหลังมีเป้าหมายเพื่อ \textbf{อธิบายความทนทานเชิงวิศวกรรมของวิธี fingerprinting} ไม่ใช่การรับรองความแม่นยำของ IPS เชิงมาตรฐานหรือเชิงคลินิก
การอ้างอิงถึงความหน่วงหรือความสดใหม่ของข้อมูลบน UI ในแบบสอบถามหมายถึงประสบการณ์ผู้ใช้ภายใต้กลไก polling และเซิร์ฟเวอร์ ไม่ใช่การรับรองทางคลินิก
\textbf{เอาต์พุต LLM} ไม่ถูกนิยามเป็นการวินิจฉัยหรือคำแนะนำทางการแพทย์

\textbf{ความเชื่อมั่นของการวัดและการทำซ้ำ (reliability / repeatability):} การทำซ้ำการประเมินสี่ประเด็นอาศัยชุดพารามิเตอร์ Profile-A (หัวข้อ~\ref{sec:ch3_eval_profile}) ร่วมกับโปรโตคอลการติดตั้งและการเตรียมระบบ (หัวข้อ~\ref{sec:ch3_protocols}) เพื่อให้สภาพแวดล้อมระหว่างรอบเก็บข้อมูลสอดคล้องกัน การตรวจสอบความพร้อมของระบบก่อนให้ผู้เข้าร่วมทำแบบประเมินหรือชุด LLM เป็นการยืนยันพื้นฐานการทำงาน ไม่ได้ขยายจำนวนชุดประเมินเชิงวิจัยนอกเหนือจากสี่ประเด็นข้างต้น

\textbf{ภัยคุกคามภายใน (internal threats):} (1) RSSI ไม่สถิติตามเวลาและสภาพแวดล้อม—จัดการด้วย QC fingerprint กฎ fallback และ failure modes ในหัวข้อ~\ref{sec:ch3_failure_modes} (2) เวลาอุปกรณ์/เซิร์ฟเวอร์ไม่สอดคล้อง—ใช้ server-side timestamp และ \texttt{seq} ตามหัวข้อ~\ref{sec:ch3_ordering} (3) การอัปเดตแบบ polling ทำให้ความสดใหม่มีขีดจำกัดตามคาบดึงข้อมูลใน Profile-A (4) เอาต์พุต LLM อาจไม่ซ้ำกันที่อุณหภูมิเดียวกัน—บันทึก log และทำซ้ำหลายรอบตามโปรโตคอล LLM/MCP

รายละเอียดเชิงทฤษฎีของ IPS และแบบจำลองสัญญาณอยู่ในบทที่~\ref{chapter2}; การตีความผลเชิงปริมาณและข้อจำกัดของการทดลองจริงอยู่ในบทที่~\ref{chapter4}

\section{โมดูลอุปกรณ์บนเก้าอี้รถเข็น}
\label{sec:ch3_wheelchair_module}

\subsection{สถาปัตยกรรมฝั่งอุปกรณ์}

โมดูลบนเก้าอี้รถเข็นออกแบบให้ทำงานเป็นหน่วย semi-autonomous โดยมีงานหลักคือการอ่านเซ็นเซอร์ การรับสัญญาณ BLE/RSSI สำหรับ localization การจัดคิวข้อมูลชั่วคราวเมื่อสัญญาณขาด และการส่งข้อมูลตามรอบเวลาที่กำหนด
การระบุตัวอุปกรณ์ใช้ \texttt{device\_id} ใน payload MQTT เพื่อให้ฝั่งเซิร์ฟเวอร์สามารถผูกข้อมูลกับการลงทะเบียนอุปกรณ์และการจับคู่ผู้ใช้/เก้าอี้รถเข็นได้อย่างเป็นระบบ
รอบการสแกน BLE เพื่อรวบรวม RSSI อาจตั้งให้ถี่กว่ารอบการส่ง telemetry รวม (เช่น สแกนเป็นคาบสั้น ๆ แล้วรวมผลเป็นชุดต่อข้อความ uplink) เพื่อลดภาระเครือข่ายโดยยังคงความถี่ของตัวอย่าง RSSI เพียงพอต่อ fingerprinting

\subsection{เซ็นเซอร์ IMU และข้อมูลการเคลื่อนไหว}

IMU บน M5StickC Plus2 ใช้สำหรับอ่านความเร่งและการเปลี่ยนแปลงทิศทางเพื่อสนับสนุนการตรวจจับรูปแบบการเคลื่อนที่
อัตราการสุ่มและช่วงการส่งข้อมูลถูกกำหนดให้สมดุลระหว่างความละเอียดกับภาระเครือข่าย โดยรายละเอียดตัวเลขอ้างอิงการตั้งค่าที่ใช้ในชุดทดสอบสรุปในหัวข้อ~\ref{sec:ch3_params}

\subsubsection{สมการประมวลผลสัญญาณการเคลื่อนที่ฝั่งอุปกรณ์ (อ้างอิงเฟิร์มแวร์)}

โค้ดฝั่งอุปกรณ์ (\texttt{SensorManager.cpp}) ปรับเทียบค่า offset ของไจโรสโคปแกน \(Z\) จากค่าเฉลี่ยในหน้าต่างเวลาเริ่มต้น แล้วใช้ตัวกรองเชิงเส้นแบบถ่วงเชิงเลขชี้ (EMA) ต่อความเร็วเชิงมุม \(\omega_Z\) (หน่วยองศาต่อวินาที) ด้วยค่าสัมประสิทธิ์ \(\alpha\) ตาม
\begin{equation}
    \hat{\omega}_{Z,t} = \alpha\,\omega_{Z,t} + (1-\alpha)\,\hat{\omega}_{Z,t-1}.
\end{equation}
หาก \(|\hat{\omega}_{Z,t}|\) ต่ำกว่าเกณฑ์ deadband \(\omega_{\mathrm{db}}\) ระบบถือว่าไม่มีการหมุนรอบแนวตั้งในมุมมองของการประมาณระยะทางในขั้นตอนนั้น
เมื่อ \(\Delta t\) เป็นช่วงเวลาระหว่างการอ่าน IMU ที่ยอมรับได้ การเปลี่ยนแปลงมุม (เรเดียน) ประมาณได้จาก \(\Delta\theta_t = \hat{\omega}^{(\mathrm{rad})}_{Z,t}\,\Delta t\) โดยแปลงจากองศาต่อวินาทีเป็นเรเดียนต่อวินาทีด้วยปัจจัย \(\pi/180\)
ให้ \(r\) เป็นรัศมีล้อที่ตั้งค่าได้ (เมตร) ระยะทางเชิงประมาณที่สะสมในช่วง \(\Delta t\) ใช้ \(\Delta s_t = |\Delta\theta_t|\, r\) แล้วรวมในหน้าต่างเวลา \(T\) (มิลลิวินาที) เพื่อได้ความเร็วเชิงเส้น
\begin{equation}
    v_t = \min\!\left(v_{\mathrm{raw},\,t},\,v_{\max}\right),\qquad
    v_{\mathrm{raw},\,t} \approx \frac{1}{T/1000}\sum \Delta s \quad \text{ภายในหน้าต่าง }T
\end{equation}
โดยมีกฎ decay เมื่อไม่ตรวจพบการเคลื่อนไหวเกินเกณฑ์เวลา การจำกัดค่าสูงสุด \(v_{\max}\) และการบังคับค่าใกล้ศูนย์ให้เป็นศูนย์เมื่อต่ำกว่าเกณฑ์
ความเร่งเชิงเส้นที่ส่งใน \texttt{motion} ประมาณจากอัตราการเปลี่ยนระหว่างความเร็วในหน้าต่างต่อเนื่อง:
\begin{equation}
    a_t = \frac{v_t - v_{t-1}}{T/1000}\quad (\mathrm{m}/\mathrm{s}^2).
\end{equation}
ค่าคงที่ \(\alpha\), \(\omega_{\mathrm{db}}\), \(T\), \(v_{\max}\) และเกณฑ์ decay ถูกกำหนดในโค้ดเฟิร์มแวร์และต้องอ้างอิงคู่กับเวอร์ชันที่ใช้ในการรายงานผล

เมื่อมีการบันทึกชุดตัวอย่าง IMU เป็นอนุกรมเวลาเพื่อสกัดฟีเจอร์ฝั่งเซิร์ฟเวอร์ (เช่น การจำแนกรูปแบบการเคลื่อนไหวด้วย XGBoost) โมดูล \texttt{feature\_engineering.py} แบ่งอนุกรมเป็นหน้าต่างความยาว \(W\) ตัวอย่างและความซ้อนทับ \(o\in[0,1)\) โดยใช้ขั้นห่าง
\begin{equation}
    s = \max\bigl(1,\lfloor W(1-o)\rfloor\bigr)
\end{equation}
ซึ่งค่าเริ่มต้นในโค้ดคือ \(W=40\) และ \(o=0.5\) (สอดคล้อง 2 วินาทีที่อัตรา 20 Hz และ stride 20 ตัวอย่าง)

\subsection{Polar Verity Sense และสัญญาณสรีระศาสตร์}

การเชื่อมต่อ Polar Verity Sense ดำเนินการภายใต้ข้อจำกัดการจับคู่อุปกรณ์และความถี่การอ่านที่เอกสารผู้ผลิตกำหนด \cite{PolarVeritySenseSDK}
ข้อมูลจากเซ็นเซอร์ทั้งสองชุดถูกนำมาใช้ควบคู่กับ IMU ในการวิเคราะห์เชิงเวลา โดยต้องคำนึงถึงความล่าช้าและการสูญเสียแพ็กเก็ตระหว่างการรวมข้อมูลบนเซิร์ฟเวอร์

\subsection{กลไกตรวจจับความผิดปกติแบบกฎเกณฑ์}
\label{sec:ch3_anomaly_rules}

ระบบใช้ตรรกะแบบกฎจากค่า accelerometer และสรุปการเคลื่อนไหว (\texttt{motion}) เป็นขั้นต่ำก่อนการนำไปสู่การแจ้งเตือน โดยกำหนดเกณฑ์เช่นการเบี่ยงเบนของแกนที่เลือกและช่วงเวลาที่ต้องเกินก่อนถือว่าเป็นเหตุการณ์
กลไกนี้มีจุดประสงค์เพื่อลดความซับซ้อนและให้ตรวจสอบได้ในระดับต้นแบบ ไม่ได้อ้างว่าเป็นมาตรฐานทางการแพทย์
ในต้นแบบนี้การตรวจจับเบื้องต้นด้วยกฎดำเนินการบนอุปกรณ์ (หรือโมดูลที่ใกล้เซ็นเซอร์ที่สุด) เพื่อลดความหน่วงก่อนแจ้งเตือน ส่วนการรวมกับตำแหน่งหรือการบันทึกเหตุการณ์ยาวขึ้นทำที่เซิร์ฟเวอร์
เกณฑ์เชิงตัวเลขที่ใช้ในชุดประเมิน/จำลองบางสถานการณ์ฝั่งเซิร์ฟเวอร์ (เช่น ช่วงอัตราการเต้นของหัวใจสูง/ต่ำ) ถูกรวมเป็นค่าเริ่มต้นในชุด \texttt{ALERT\_THRESHOLDS} ของ \texttt{sim\_controller.py} ในโครงงาน \texttt{wheelsense-platform} สำหรับการทดสอบที่เปิดใช้งานการจำลอง ส่วนเกณฑ์จาก IMU บนอุปกรณ์จริงเป็นพารามิเตอร์ที่ปรับได้และต้องอ้างอิงคู่กับเวอร์ชันเฟิร์มแวร์ (เช่น \texttt{FIRMWARE\_VERSION} ในโค้ดอุปกรณ์) ในเอกสารประกอบหรือภาคผนวกเพื่อการทำซ้ำ

\subsection{การสื่อสารไร้สายและการรายงานสถานะ}

การส่ง telemetry ใช้โปรโตคอลที่เหมาะกับอุปกรณ์ฝังตัวและโครงสร้าง MQTT บนเซิร์ฟเวอร์ \cite{MQTTSpec}

เมื่อการเชื่อมต่อขาด อุปกรณ์เข้าสู่โหมดลองเชื่อมต่อใหม่และคิวข้อมูลชั่วคราวตามความจำเป็น แล้วส่งย้อนหลังหลังกลับมาออนไลน์พร้อมสถานะช่องว่างของข้อมูลเพื่อให้ผู้ดูแลทราบ

\subsection{โครงข่าย BLE และการรวบรวม RSSI}

BLE beacon ถูกจัดวางให้ครอบคลุมพื้นที่อ้างอิงและโซนที่ต้องการแยกความละเอียดในการประเมิน โดย M5StickC Plus2 บนเก้าอี้รถเข็นเป็นตัวรับสัญญาณ BLE/RSSI ตามรอบเวลาที่ตั้งไว้และส่งเวกเตอร์หรือค่าที่ใช้ประกอบ fingerprinting ขึ้นเซิร์ฟเวอร์
ปัจจัยที่มีผลต่อคุณภาพสัญญาณอธิบายไว้ในวรรณกรรม IPS และไม่ถูกทวนซ้ำเชิงทฤษฎีในบทนี้ \cite{BLE_Fingerprint_Survey_IEEE,WiFi_BLE_IPS_Systematic_Review}

\subsection{การสร้าง fingerprint และแผนที่สัญญาณ}

\textbf{ขั้นตอนออฟไลน์ (สร้างแผนที่อ้างอิง)} มีเป้าหมายเพื่อสร้าง radio map ที่ผูกเวกเตอร์ RSSI กับป้ายกำกับโซน/ห้องอย่างเป็นระบบ โดยกำหนดโปรโตคอลให้ทำซ้ำได้ดังนี้
\begin{enumerate}
    \item \textbf{กำหนดขอบเขตพื้นที่และระดับความละเอียด} — นิยามโซน/ห้องที่ต้องการทำนาย และกำหนดว่า output จะรายงานเป็น room-level หรือ zone-level ให้สอดคล้องกับการประเมินในบทที่~\ref{chapter4}
    \item \textbf{กำหนดจุดอ้างอิง (reference points)} — ภายในแต่ละโซน/ห้องกำหนดจุดอ้างอิงให้ครอบคลุมพื้นที่ใช้งานจริง เช่น ทางเดิน จุดเลี้ยว จุดหน้าประตู และจุดที่มีสิ่งกีดขวาง โดยจำนวนจุดอ้างอิงและระยะห่างระหว่างจุดถูกกำหนดเป็นพารามิเตอร์ (หัวข้อ~\ref{sec:ch3_params})
    \item \textbf{การเก็บข้อมูลต่อจุด} — เคลื่อนที่เก้าอี้ไปยังจุดอ้างอิงและคงตำแหน่งตามระยะเวลาที่กำหนดเพื่อเก็บหลายตัวอย่าง ลดผลจากการสแกนเพียงครั้งเดียวที่อาจผิดปกติจากการสะท้อนสัญญาณ
    \item \textbf{การติดป้ายกำกับ (labeling)} — ทุกชุดข้อมูลต้องมีป้ายกำกับโซน/ห้องของจุดอ้างอิง รวมถึง metadata สำคัญ เช่น ช่วงเวลา และสภาพแวดล้อมที่ส่งผลต่อ RSSI (เช่น ประตูเปิด/ปิด คนเดินผ่าน) เพื่อรองรับการอภิปรายความเสถียรของระบบ
    \item \textbf{การตรวจสอบคุณภาพข้อมูล (QC)} — ตรวจสอบความครบถ้วนของโหนด/บีคอนที่ถูกสแกน ความสม่ำเสมอของจำนวนตัวอย่างต่อจุด และคัดทิ้งตัวอย่างที่อยู่นอกช่วงที่เป็นไปได้หรือมีความผิดปกติที่ชัดเจนตามกฎเดียวกันทั้งชุด
    \item \textbf{การรวมเป็น radio map} — รวมข้อมูลทั้งหมดเป็นโครงสร้างอ้างอิงในฐานข้อมูล โดยกำหนด schema และเวกเตอร์ฟีเจอร์ให้คงที่สำหรับการฝึก/ทำนายในขั้นตอนออนไลน์
\end{enumerate}
การเก็บ RSSI ณ จุดอ้างอิงใช้ข้อมูลที่อุปกรณ์บนเก้าอี้รับและส่งขึ้นมา ไม่ใช้สตรีมภาพจาก ESP32-S3 ในขั้นตอนนี้
หากสภาพแวดล้อมเปลี่ยน (เช่น การย้ายสิ่งของ การเปลี่ยนตำแหน่งบีคอน หรือสภาพคนพลุกพล่านเปลี่ยนถาวร) ระบบกำหนดให้เก็บ fingerprint ใหม่หรือปรับน้ำหนักข้อมูลตามโปรโตคอลเดียวกันเพื่อรักษาความเที่ยงตรงของการทำนาย

\subsection{การประมาณตำแหน่งแบบออนไลน์ด้วย KNN}

\textbf{ขั้นตอนออนไลน์} เซิร์ฟเวอร์รับ snapshot ของ RSSI ล่าสุดจาก M5 ผ่าน MQTT แล้วทำการจัดรูปแบบหรือกรองค่าที่ขาดหาย/ผิดปกติตามนโยบายที่ตั้งไว้ จากนั้นเปรียบเทียบกับ fingerprint ในฐานข้อมูลด้วยระยะห่างที่กำหนดและค่า \(k\) ของ KNN ตามหลักการในบทที่~\ref{chapter2} เพื่อได้ผลลัพธ์เป็นโซนหรือห้อง
ภาพจากโหนดกล้องใช้ประกอบการรับรู้บริบทและการแสดงผลต่อผู้ดูแล ไม่ได้บูรณาการเป็นฟีเจอร์ใน KNN ในต้นแบบนี้
การเลือก \(k\) และนิยามฟีเจอร์ RSSI เป็นพารามิเตอร์สำคัญที่สรุปในหัวข้อ~\ref{sec:ch3_params} และต้องเป็นชุดเดียวกับที่ใช้ในการรายงานผลในบทที่~\ref{chapter4}

\subsubsection{สมการจับคู่ KNN บนเซิร์ฟเวอร์ (สอดคล้อง scikit-learn)}

การฝึกและทำนายในต้นแบบใช้ \texttt{sklearn.neighbors.KNeighborsClassifier} โดยกำหนด \texttt{metric='euclidean'} และ \texttt{weights='distance'} ให้เวกเตอร์อินพุต \(\mathbf{x}\in\mathbb{R}^{d}\) มีมิติตามลำดับโหนด \texttt{node\_order} และค่าขาดถูกแทนด้วย sentinel (เช่น \(-100\) dBm) ตามหัวข้อ~\ref{sec:ch3_feature_vector}
ระยะห่างยูคลิดระหว่างตัวอย่างทดสอบ \(\mathbf{x}\) กับตัวอย่างฝึก \(\mathbf{x}^{(i)}\) คือ
\begin{equation}
    d(\mathbf{x},\mathbf{x}^{(i)}) = \sqrt{\sum_{j=1}^{d}\bigl(x_j - x^{(i)}_{j}\bigr)^2}.
\end{equation}
เมื่อใช้การถ่วงน้ำหนักตามระยะทาง ค่าน้ำหนักของเพื่อนบ้าน \(i\) อยู่ในสัดส่วน \(1/d(\mathbf{x},\mathbf{x}^{(i)})\) (กรณี \(d=0\) ถูกจัดการในไลบรารีให้ไม่หารศูนย์) ซึ่งสอดคล้องกับการให้เพื่อนบ้านที่ใกล้กว่ามีอิทธิพลมากกว่าในการจำแนกป้ายห้อง/โซน
ค่า \(k\) ที่ใช้จริงถูกจำกัดด้วย \(\min(5, n_{\mathrm{samples}})\) ในกระบวนการฝึกเมื่อจำนวนตัวอย่างฝึกมีจำกัด ตามการตั้งค่าในโมดูล \texttt{localization.py} ของโครงการ

\subsection{นิยามเวกเตอร์ฟีเจอร์และการจัดการสัญญาณที่ขาดหาย}
\label{sec:ch3_feature_vector}

เพื่อให้การฝึกและการทำนายของ KNN มีความสอดคล้อง เวกเตอร์ฟีเจอร์ RSSI ถูกนิยามบนชุดมิติ (dimensions) ที่สอดคล้องกับรายการโหนด/บีคอนที่ระบบรับรู้ในพื้นที่ทดสอบ โดยกำหนดลำดับมิติให้คงที่ (node ordering) และใช้กฎเดียวกันทั้งในช่วงออฟไลน์และออนไลน์
การแปลงจาก \texttt{rssi[]} ไปสู่เวกเตอร์ทำเป็นขั้นตอนมาตรฐาน ได้แก่ (1) กลุ่มข้อมูลตามโหนด (2) เลือกค่าตัวแทนต่อโหนดในกรณีมีหลายรายการ (เช่น ค่ามัธยฐานในหน้าต่างเวลา) (3) เติมค่ามิติที่หายไปตามนโยบาย missing-signal

ในกรณีที่รอบสแกนหนึ่งไม่พบสัญญาณจากบางโหนด ระบบใช้นโยบายแทนค่า (missing-signal handling) เพื่อให้เวกเตอร์มีความยาวคงที่และทำงานกับโมเดลได้ โดยนโยบายที่ใช้ในต้นแบบกำหนดเป็นพารามิเตอร์ เช่น
\begin{enumerate}
    \item \textbf{Sentinel dBm} — แทนค่าที่ขาดด้วยค่าต่ำกว่า RSSI ที่พบได้จริง (เช่น \(-100\) dBm) เพื่อสะท้อน “ไม่พบสัญญาณ”
    \item \textbf{Last-observation carry-forward} — คงค่าจากรอบก่อนหน้าในหน้าต่างเวลาสั้น ๆ เมื่อคาดว่าการหายไปเป็นความผันผวนระยะสั้น
    \item \textbf{Mask/weighting} — ใช้ตัวบ่งชี้การขาดหายหรือถ่วงน้ำหนักมิติที่ขาด เพื่อไม่ให้ sentinel ครอบงำระยะห่าง
\end{enumerate}
\textbf{ใน Profile-A} การรายงานผลในบทที่~\ref{chapter4} ใช้ \textbf{sentinel} ที่ \(-100\) dBm เป็นหลักสำหรับมิติที่ขาด พร้อม \textbf{last-known hold} ตามตารางที่~\ref{tab:ch3_profileA_loc} ไม่ได้ใช้ mask/weighting แบบแยกมิติในการสรุปผลหลัก
ระบบเลือกใช้นโยบายเพียงแบบเดียวเป็นค่าอ้างอิงในการประเมินจริง และบันทึกไว้ในหัวข้อ~\ref{sec:ch3_params} เพื่อให้บทที่~\ref{chapter4} ทำซ้ำได้

\subsection{กลยุทธ์สำรองและ baseline เชิงปฏิบัติการ (max-RSSI)}
\label{sec:ch3_max_rssi}

เพื่อให้ระบบยังคงให้บริการได้เมื่อโมเดล KNN ยังไม่พร้อม (เช่น หลังเปลี่ยนพื้นที่ทดลองหรือยังไม่ฝึกโมเดลจาก fingerprint ล่าสุด) ระบบกำหนดกลยุทธ์ max-RSSI เป็น baseline และ fallback เชิงปฏิบัติการ
แนวทาง max-RSSI ใช้หลักการเลือกโหนด/บีคอนที่มี RSSI สูงสุดในรอบสแกน แล้วแมปไปยังโซน/ห้องตามตารางผูกความสัมพันธ์ของพื้นที่ (room-to-node binding) เพื่อให้ระบบยังคงให้ตำแหน่งโดยประมาณแบบใกล้เวลาจริง
หากไม่พบสัญญาณหรือค่ามีความไม่แน่นอนสูง ระบบทำเครื่องหมายความน่าเชื่อถือต่ำและเลือกคงค่าตำแหน่งล่าสุดตามนโยบายที่กำหนด เพื่อลดการกระโดดของตำแหน่งและการแจ้งเตือนผิดพลาด

\subsection{กรณีขอบเขตของข้อมูลและข้อจำกัด}

\subsection{Pipeline ข้อมูลแบบเรียลไทม์}

ลำดับการประมวลผลหลักประกอบด้วยการรับข้อมูลจากอุปกรณ์และโหนดผ่าน MQTT การตรวจสอบความถูกต้องเบื้องต้น การบันทึกลงฐานข้อมูล และการเผยแพร่ต่อ API/ช่องทางอัปเดตแบบใกล้เวลาจริงตามที่ออกแบบในแอปพลิเคชัน
สตรีมภาพอาจส่งเป็นชิ้นส่วนภายใต้หัวข้อเฉพาะของโหนดกล้อง โดยบริการ ingestion ประกอบ metadata (เช่น ลำดับแพ็กเก็ต ความละเอียด) ก่อนจัดเก็บหรือส่งต่อไปยัง buffer ชั่วคราวสำหรับการแสดงผล
จุดที่ต้องบันทึกเวลาอย่างสม่ำเสมอคือขั้นตอนรับเข้าและก่อนบันทึกลงฐานข้อมูล เพื่อให้การวิเคราะห์ย้อนหลังสอดคล้องกับลำดับเหตุการณ์

\subsubsection{ขั้นตอน ingestion/validation และกฎความถูกต้องขั้นต่ำ}
\label{sec:ch3_ingestion_validation}

เพื่อให้ข้อมูลที่เข้าสู่ฐานข้อมูลมีความสอดคล้องและตรวจสอบย้อนหลังได้ บริการ ingestion กำหนดขั้นตอนมาตรฐานดังนี้
\begin{enumerate}
    \item \textbf{ตรวจสอบตัวระบุและบริบท} — ตรวจสอบว่ามี \texttt{device\_id} และสามารถผูกกับการลงทะเบียนอุปกรณ์ใน workspace ที่ถูกต้องก่อนบันทึกข้อมูล เพื่อลดความเสี่ยงจากข้อมูลหลุดขอบเขต (cross-workspace contamination)
    \item \textbf{ตรวจสอบ schema ขั้นต่ำ} — ตรวจสอบชนิดข้อมูลและฟิลด์ที่จำเป็นของ payload (เช่น \texttt{imu} เป็น object, \texttt{rssi[]} เป็น array, \texttt{seq} เป็นจำนวนเต็ม) พร้อมบันทึกเหตุผลการปฏิเสธเมื่อ payload ไม่ถูกต้อง
    \item \textbf{ทำ normalization} — แปลงหน่วย/รูปแบบให้สอดคล้อง (เช่น RSSI เป็น dBm, สถานะชาร์จเป็น boolean) และจัดรูปแบบเวลาให้อยู่ในรูปเดียวกัน
    \item \textbf{ประทับเวลา (server-side timestamp)} — เพิ่มเวลาเมื่อรับเข้าและเวลาเมื่อบันทึก เพื่อใช้คำนวณ latency และตรวจสอบ out-of-order โดยไม่ขึ้นกับการ sync เวลาในอุปกรณ์เพียงอย่างเดียว
    \item \textbf{บันทึก time-series และเหตุการณ์} — แยกการบันทึกเป็นหมวด เช่น telemetry, RSSI snapshot, prediction, alert/event และ media metadata เพื่อให้สืบค้นได้มีประสิทธิภาพ
\end{enumerate}

\subsubsection{การจัดการลำดับข้อมูลและกรณี out-of-order}
\label{sec:ch3_ordering}

ระบบใช้ \texttt{seq} และ \texttt{uptime\_ms} จากอุปกรณ์ร่วมกับ server-side timestamp เพื่อช่วยจัดการกรณีข้อมูลมาถึงไม่เรียงเวลา โดยแนวทางหลักคือ (1) ใช้เวลาเซิร์ฟเวอร์เป็นอ้างอิงสำหรับการแสดงผล “สถานะล่าสุด” (2) ใช้ \texttt{timestamp} ของอุปกรณ์เมื่อมีการ sync เพื่อการวิเคราะห์เชิงเหตุการณ์/เส้นเวลา และ (3) ใช้ \texttt{seq} ตรวจสอบการหลุดหายของข้อมูลเพื่อการอภิปรายความเสถียรของ pipeline ในบทที่~\ref{chapter4}

\subsubsection{การประกอบคืนภาพ (chunk reassembly) และความทนทานต่อความผิดพลาด}
\label{sec:ch3_image_reassembly}

การส่งภาพแบบ chunked payload มีความเสี่ยงต่อการสูญหายของชิ้นส่วนและการประกอบคืนไม่สำเร็จ ระบบจึงกำหนดขั้นตอนการจัดการดังนี้
\begin{enumerate}
    \item บันทึก metadata ต่อภาพอย่างน้อย: \texttt{device\_id}, ลำดับชิ้นส่วน, จำนวนชิ้นส่วนที่คาดหวัง, ความละเอียด/รูปแบบ, เวลาเริ่มรับ และเวลาเสร็จสิ้น
    \item ตั้ง timeout สำหรับภาพที่ประกอบไม่ครบภายในเวลาที่กำหนด และบันทึกเหตุการณ์ “ประกอบไม่สำเร็จ” เพื่อใช้วิเคราะห์อัตราความสำเร็จของเส้นทางภาพ
    \item เมื่อเกิดความผิดพลาดซ้ำ ให้โหนดสลับใช้ช่องทาง fallback (\texttt{.../frame}) หรือปรับพารามิเตอร์อัตรา/ความละเอียดตามที่นิยามไว้ในหัวข้อ~\ref{sec:ch3_params}
\end{enumerate}

\subsubsection{การเผยแพร่สู่ผู้ใช้และกลไกอัปเดตแบบใกล้เวลาจริง}
\label{sec:ch3_realtime_to_clients}

เพื่อให้ระบบทนทานต่อเครือข่ายที่แปรผันและทำซ้ำได้ง่าย ระบบกำหนดให้ web dashboard และ mobile app ใช้การดึงข้อมูลแบบช่วงเวลา (polling) สำหรับ endpoint สถานะล่าสุดและรายการแจ้งเตือน โดยบันทึกคาบการดึงข้อมูลเป็นพารามิเตอร์อ้างอิง (หัวข้อ~\ref{sec:ch3_params})
สำหรับชั้น MCP ที่ใช้กับ LLM ระบบรองรับการสตรีมผลแบบ HTTP streaming/SSE-compatible ตามรูปแบบโปรโตคอล MCP \cite{ModelContextProtocol} เพื่อให้สามารถติดตามลำดับการเรียกใช้เครื่องมือและเวลาแฝงรายขั้นตอนได้อย่างชัดเจน

\subsubsection{นโยบายการเก็บรักษาข้อมูล (retention) และบันทึกระบบ (logging)}
\label{sec:ch3_retention}

ระบบกำหนดนโยบาย retention สำหรับข้อมูลที่เป็น time-series และ media เพื่อควบคุมพื้นที่จัดเก็บและลดความเสี่ยงด้านข้อมูลอ่อนไหว โดยแยกหน้าต่างเวลาตามชนิดข้อมูล เช่น telemetry/RSSI/predictions/logs และกำหนดให้การลบข้อมูลเป็นแบบตรวจสอบย้อนกลับได้ (audit) เพื่อให้สอดคล้องกับการประเมินในบทที่~\ref{chapter4} และข้อกำหนดจริยธรรมในหัวข้อ~\ref{sec:ch3_privacy}

\subsection{การติดตั้งและการกำหนดค่า}

การติดตั้งใช้แพ็กเกจคอนเทนเนอร์และสคริปต์กำหนดค่าพื้นฐานเพื่อลดความคลาดเคลื่อนระหว่างรอบการทดลอง \cite{Boettiger2015_Docker_reproducible_research}
รายการพอร์ต ตัวแปรสภาพแวดล้อม และเวอร์ชันบริการที่ใช้ในชุดทดสอบควรสอดคล้องกับภาคผนวกหรือเอกสารประกอบโครงงานเพื่อการทำซ้ำ

การแสดงผลแบบใกล้เวลาจริงเชื่อมกับ API โดยใช้การดึงข้อมูลแบบช่วงเวลา (polling) ตามคาบที่กำหนดสำหรับประเภทข้อมูลที่ต่างกัน เช่น สถานะตำแหน่งล่าสุด รายการแจ้งเตือน และสรุปสัญญาณสำคัญ
ตัวอย่างการจับคู่: การอัปเดตโซน/ห้องใช้การดึง “สถานะล่าสุด” ตามคาบสั้นเพื่อให้ผู้ดูแลเห็นการเปลี่ยนแปลงทันเวลา; การแจ้งเตือนจากกฎบนอุปกรณ์หรือจากเซิร์ฟเวอร์ถูกบันทึกเป็นรายการเหตุการณ์และถูกดึงมาแสดงในรูปแบบ banner/notification panel
เมื่อการเชื่อมต่อขาด ระบบควรแสดงสถานะการเชื่อมต่อและช่วงเวลาที่ข้อมูลล่าสุดถูกอัปเดต

งานหลักของผู้ดูแลในระบบต้นแบบได้แก่ การติดตามตำแหน่งหรือโซนล่าสุด การรับทราบหรือปิดการแจ้งเตือน (acknowledge) การตรวจสอบเหตุการณ์ความผิดปกติจากกฎ และการเปิดดูประวัติ telemetry หรือเหตุการณ์ย้อนหลังเบื้องต้น
แต่ละงานเชื่อมกับเหตุการณ์จาก API และข้อมูลที่ดึงมาแบบช่วงเวลา โดยไม่จำเป็นต้องมีภาพหน้าจอในบทนี้
รายละเอียดการจัดวางเมนูและป้ายกำกับควรสอดคล้องกับรุ่นระบบที่ใช้ในช่วงประเมินสุดท้าย

\section{แอปพลิเคชันมือถือ}
\label{sec:ch3_mobile}

ตามขอบเขตในบทที่~\ref{chapter1} แอปพลิเคชันมือถือพัฒนาด้วย React Native + Expo ในฐานะช่องทางเสริมสำหรับผู้ใช้ที่ไม่ได้นั่งเก้าอี้รถเข็นและกรณีผู้ใช้ที่ต้องการโหมดการใช้งานนอกเก้าอี้ โดยรองรับการสแกน BLE/RSSI และการเชื่อมต่อ Polar แล้วส่ง telemetry ผ่าน MQTT ตามหัวข้อ~\ref{sec:ch3_mqtt_mobile}
บทบาทหลักในการติดตามแบบเรียลไทม์ยังคงอยู่ที่แอปพลิเคชันเว็บ ดังนั้นการทดสอบการใช้งานจึงลำดับความสำคัญให้สอดคล้องกับบทบาทดังกล่าว
ฝั่งมือถือใช้ชุด API และการยืนยันตัวตนเดียวกับแอปเว็บ (โดเมนของบริการ FastAPI เดียวกัน) เพื่อให้สถานะล่าสุด รายการแจ้งเตือน และงานที่เกี่ยวข้องกับผู้ดูแลสอดคล้องกันข้ามช่องทาง
การอัปเดตข้อมูลใช้แนวทางดึงเป็นระยะ (polling) หรือการรีเฟรชตามคาบที่กำหนดในการออกแบบแอป โดยอิงช่วงเวลาเชิงปฏิบัติการเดียวกับหลักการของ dashboard (หัวข้อ~\ref{sec:ch3_webapp} และพารามิเตอร์ Profile-A ในหัวข้อ~\ref{sec:ch3_params}) เพื่อให้การประเมิน UX ระหว่างเว็บและมือถือเปรียบเทียบกันได้เป็นธรรม

\section{Home Assistant และระบบอัตโนมัติ}
\label{sec:ch3_ha}

\subsection{การเชื่อมต่อและข้อมูลที่แลกเปลี่ยน}

Home Assistant เชื่อมต่อกับ WheelSense ผ่านกลไกที่เหมาะสมกับแพลตฟอร์ม เช่น MQTT integration หรือ API ที่เผยแพร่สถานะเป็น entity
เหตุการณ์ที่เผยแพร่ไปยัง HA จัดเป็นกลุ่มเชิงบทบาท ได้แก่ \textbf{การเปลี่ยนโซน/ห้อง} (สถานะตำแหน่งล่าสุด) \textbf{การแจ้งเตือนหรือ anomaly} จากกฎบนอุปกรณ์หรือจากเซิร์ฟเวอร์ และ \textbf{สถานะการเชื่อมต่อ} ของอุปกรณ์หรือโหนดกล้อง
ข้อมูลที่ข้ามขอบเขตควรจำกัดเฉพาะสถานะและเหตุการณ์ที่จำเป็นต่อ automation เพื่อลดพื้นที่การโจมตีและความเสี่ยงด้านความเป็นส่วนตัว

\subsection{รูปแบบ automation ตามบริบท}

ตัวอย่างรูปแบบ automation ได้แก่ การปรับอุปกรณ์ในบ้านเมื่อ entity โซนตรงกับค่าที่กำหนด หรือเมื่อเกิดเหตุการณ์แจ้งเตือนจากกฎบนอุปกรณ์ หรือเมื่อสถานะการเชื่อมต่อเปลี่ยนเป็นออฟไลน์
รายละเอียด entity และ YAML ควรบันทึกเป็นภาคผนวกหรือคู่มือการติดตั้งเพื่อการทำซ้ำ ไม่บรรจุ YAML เต็มในเนื้อหาหลักของบทนี้

\subsection{ความปลอดภัยและการตั้งค่า}

การควบคุมอุปกรณ์จริงในระบบบ้านอัจฉริยะมีความเสี่ยงหาก credential รั่วไหลหรือ automation ถูกกำหนดผิดพลาด ดังนั้นการทดลองควรจำกัดสิทธิ์การเข้าถึงและแยกสภาพแวดล้อมทดสอบออกจากระบบที่ใช้ในบ้านจริงเมื่อเป็นไปได้

\section{Local LLM และ MCP}
\label{sec:ch3_llm_mcp}

\subsection{บทบาทและขอบเขตการใช้งาน}

โมดูล local LLM ใช้ Gemma 4B ตามขอบเขตในบทที่~\ref{chapter1} โดยเชื่อมต่อผ่าน MCP เพื่อให้โมเดลเรียกใช้เครื่องมือที่อนุญาตเท่านั้น \cite{GoogleGemma,ModelContextProtocol}
ขอบเขตการใช้งานต้องชัดเจนว่าเป็นการช่วยสรุปและชี้ประเด็น ไม่ใช่การวินิจฉัยหรือคำแนะทางการแพทย์

\subsection{รูปแบบการทำงานแบบยืนยันก่อนกระทำ (Propose--Confirm--Execute) และโหมดการจัดเส้นทาง}

เพื่อให้การกระทำที่เปลี่ยนแปลงสถานะ (write actions) มีความปลอดภัยและตรวจสอบย้อนหลังได้ ระบบกำหนดการทำงานเป็น 3 ระยะ ได้แก่ \textbf{เสนอ (propose)} $\rightarrow$ \textbf{ยืนยัน (confirm)} $\rightarrow$ \textbf{ดำเนินการ (execute)} กล่าวคือ ระบบสามารถตอบคำถามหรือเรียกใช้เครื่องมือประเภทอ่านข้อมูล (read tools) เพื่อสร้างคำตอบเชิงหลักฐานได้ทันที แต่เมื่อเป็นเครื่องมือที่มีผลต่อข้อมูลหรือการควบคุมอุปกรณ์ ระบบต้องสร้างแผนการทำงานเพื่อให้ผู้ใช้ยืนยันก่อนจึงดำเนินการจริง
นอกจากนี้ระบบรองรับโหมดการจัดเส้นทางคำขอ 2 แบบ ได้แก่ (1) \textbf{intent-based routing} ที่ใช้การจำแนกเจตนาเพื่อเลือกเครื่องมือ และ (2) \textbf{LLM tool routing} ที่ให้โมเดลเลือกเครื่องมือจากรายการที่อนุญาตภายใต้ข้อจำกัดสิทธิ์ โดยทั้งสองโหมดอยู่ภายใต้กรอบการยืนยันก่อนกระทำเดียวกัน

\subsection{เครื่องมือและแหล่งข้อมูล}

เครื่องมือ MCP จัดเป็นประเภทเชิงอินเทอร์เฟซ ได้แก่ \textbf{การอ่านสถานะ} (เช่น สรุป telemetry โซนล่าสุด หรือข้อความจากเซิร์ฟเวอร์) และ \textbf{การกระทำที่จำกัด} (เช่น คำสั่งที่ผูกกับ entity ของ Home Assistant หรือการทริกเกอร์ที่อนุญาต) โดยไม่เปิด endpoint ทั่วไป
การออกแบบนี้ลดความเสี่ยงจากการเรียกใช้ฟังก์ชันที่ไม่คาดคิดและสอดคล้องกับแนวทางการประเมินความน่าเชื่อถือของเอาต์พุตภาษา \cite{Ji2023HallucinationSurvey}

\subsection{การทำงานร่วมกับแอปพลิเคชันและเซิร์ฟเวอร์}

ลำดับการทำงานทั่วไปเริ่มจากคำขอของผู้ใช้หรือเหตุการณ์ในระบบ ไปสู่การดึงข้อมูลอ้างอิง การเรียกเครื่องมือ และการคืนค่าผลลัพธ์ไปยังชั้นแสดงผล
การบันทึก log ควรเก็บอย่างน้อย: เวลาที่ได้รับคำขอ รายการเครื่องมือที่ถูกเลือก พารามิเตอร์ที่ส่งเข้าเครื่องมือ เวลาที่เครื่องมือตอบกลับสำเร็จ และข้อความสรุปของโมเดล เพื่อให้สอดคล้องกับนิยามการประเมินในหัวข้อ~\ref{sec:ch3_eval_llm_metrics} และ~\ref{sec:ch3_eval_irr} โดยตัวเลขและการอภิปรายนำเสนอในบทที่~\ref{chapter4}

\section{ความปลอดภัย ความเป็นส่วนตัว และจริยธรรมข้อมูล}
\label{sec:ch3_privacy}

ข้อมูลในระบบแบ่งได้เป็นชั้นหลัก ได้แก่ \textbf{ตำแหน่งและพฤติกรรมการเคลื่อนที่} \textbf{สัญญาณสรีระศาสตร์} และ \textbf{ภาพจากกล้อง} ซึ่งล้วนมีความอ่อนไหวในเชิงการใช้งานและบริบทสถานที่
การออกแบบต้นแบบกำหนดการควบคุมการเข้าถึงแอปพลิเคชันและบทบาทผู้ใช้ การส่งผ่านข้อมูลภายในเครือข่ายที่ควบคุมได้ และนโยบายการเก็บรักษาและลบข้อมูลภาพหรือ log หลังการทดลองตามความจำเป็น
ขอบเขตการเก็บรักษาเป็นระดับต้นแบบในงานวิจัย ไม่ได้อ้างมาตรฐานความปลอดภัยระดับผลิตภัณฑ์ โดยอิงหลักการด้านจริยธรรมและความเป็นส่วนตัวที่กรอบไว้ในบทที่~\ref{chapter2} \cite{Wilkowska2022AAL_privacy_acceptance}

\section{พารามิเตอร์และการตั้งค่าหลัก}
\label{sec:ch3_params}

พารามิเตอร์ในระบบถูกแบ่งออกเป็น 2 กลุ่มเพื่อความชัดเจน ได้แก่ (1) พารามิเตอร์ด้าน localization และการเก็บ fingerprint (2) พารามิเตอร์ด้านอุปกรณ์ เครือข่าย ระบบบริการ และ LLM/MCP ทั้งสองกลุ่มต้องใช้ชุดค่าที่สอดคล้องกับการประเมินสี่ประเด็นและการรายงานผลในบทที่~\ref{chapter4}
หากมีการปรับค่าระหว่างพัฒนา ให้บันทึกชุดค่าที่ใช้ในการประเมินจริงเป็นชุดเดียวและเก็บหลักฐานการตั้งค่าไว้ในภาคผนวกเพื่อการทำซ้ำ

\subsection{Evaluation profile สำหรับการรายงานผลในบทที่ 4}
\label{sec:ch3_eval_profile}

ระบบถูกพัฒนาให้ปรับพารามิเตอร์ได้ (configurable) เพื่อรองรับสภาพแวดล้อมที่ต่างกัน อย่างไรก็ตามเพื่อให้การประเมินสี่ประเด็นและผลในบทที่~\ref{chapter4} ทำซ้ำได้อย่างชัดเจน โครงงานกำหนด \textbf{evaluation profile} เดียวเป็นค่าอ้างอิง เรียกว่า \textbf{Profile-A}
โดยการรายงานตัวเลขที่อ้างอิงสภาพระบบในบทที่~\ref{chapter4} ใช้ค่าตาม Profile-A เท่านั้น ส่วนการตั้งค่าอื่นที่ระบบรองรับไม่ถูกนำมาขยายขอบเขตการอภิปรายเกินความจำเป็น

เพื่อให้การอ้างอิง Profile-A ตรวจสอบย้อนหลังได้ ชุดค่าที่ใช้คู่กับการรายงานผลควรถูก ``แช่แข็ง'' โดยเก็บหลักฐานอย่างน้อยดังนี้: (1) ตัวระบุเวอร์ชันของที่เก็บโค้ด \texttt{wheelsense-platform} (เช่น แฮชคอมมิตหรือแท็กรุ่น) ณ ช่วงทดลอง (2) ไฟล์กำหนดค่าคอนเทนเนอร์/ตัวแปรสภาพแวดล้อมที่ใช้บน Raspberry Pi (3) เวอร์ชันเฟิร์มแวร์อุปกรณ์ที่อ่านได้จากข้อความ telemetry หรือจากเอกสารกำหนดค่า (4) สำเนาการตั้งค่า MQTT ที่ใช้ในห้องทดลอง (เช่น \texttt{mosquitto.conf}) เมื่อเกี่ยวข้องกับพอร์ตและนโยบายการยืนยันตัวตน

\subsection{นิยามข้อเท็จจริงอ้างอิง (ground truth) และการจัดแนวเชิงเวลา}
\label{sec:ch3_ground_truth}

\textbf{Ground truth สำหรับ localization (ระดับโซน/ห้อง):} ในช่วงเก็บ fingerprint ป้ายกำกับ \(y\) ของแต่ละตัวอย่างคือโซน/ห้องที่จุดอ้างอิง (RP) ตั้งอยู่จริงบนผังพื้นที่ ไม่ใช่การตีความจากระบบ
ในช่วงทดสอบออนไลน์หรือการทดสอบความทนทานแบบจำลองสัญญาณรบกวน ป้ายกำกับ \(y\) ยังคงเป็นห้อง/โซนที่ผู้ทดลองควบคุมให้อยู่จริงในขณะเก็บข้อมูล แม้เวกเตอร์ \(\tilde{\mathbf{r}}\) จะถูกปรับให้ ``คล้ายแต่ไม่เหมือนเดิม'' ตามหัวข้อ~\ref{sec:ch3_eval_loc_noise_injection}

\textbf{การจัดแนวเชิงเวลา:} เมื่อรวมสตรีมหลายแหล่ง (เช่น telemetry กับ prediction กับ alert) ให้ใช้ \texttt{server\_receive\_ts} เป็นหลักสำหรับการเรียงเหตุการณ์ในการวิเคราะห์ end-to-end ตามหลักการในบทที่~\ref{sec:ch2_multistream} และใช้ \texttt{device\_timestamp} ร่วมกับ \texttt{seq} เมื่อต้องการวิเคราะห์พฤติกรรมฝั่งอุปกรณ์
หากต้องเทียบเหตุการณ์กับกิจกรรมของผู้ทดลอง ให้กำหนดหน้าต่างยอมรับ \(\Delta t\) (เช่น 1--2 คาบ telemetry ตาม Profile-A) รอบจุดเปลี่ยนโซนในสคริปต์ทดสอบ เพื่อลดความคลาดเคลื่อนจากความหน่วงของเครือข่าย

\subsubsection{ขั้นตอนปฏิบัติ}
\begin{enumerate}
    \item ยืนยันว่าตำแหน่ง BLE beacon ไม่เปลี่ยนจากแผนผังพื้นที่ทดสอบ
    \item กำหนดจุดอ้างอิงในแต่ละโซน/ห้องตามระยะห่างและจำนวนที่กำหนดใน Profile-A
    \item สำหรับแต่ละจุด: จอดนิ่งตาม dwell time, เก็บข้อมูลครบตามจำนวนรอบ, ใส่ป้ายกำกับโซน/ห้องและ metadata สภาพแวดล้อม
    \item หลังจบโซน: ตรวจ QC เบื้องต้น (coverage, outlier, จำนวนตัวอย่าง) ก่อนย้ายไปโซนถัดไป
    \item รวมชุดข้อมูลทั้งหมดเป็น radio map และบันทึกเวอร์ชันของชุด fingerprint ที่ใช้ในการฝึกโมเดล
\end{enumerate}

\section{แผนการประเมินผล}
\label{sec:ch3_test_plan}

หัวข้อนี้กำหนด \textbf{การประเมินผลเชิงวิจัย} ของงาน โดยประกอบด้วย (ก) \textbf{สี่ประเด็นด้าน UX/UI และ LLM/MCP} และ (ข) \textbf{การทดสอบเสริมด้านความทนทานของโมดูลระบุตำแหน่งต่อความผันผวนของ RSSI} แบบจำลอง (Gaussian noise injection) เพื่อเปรียบเทียบ KNN กับ XGBoost ในบริบทข้อมูลเดียวกัน
การตรวจสอบการทำงานของอุปกรณ์ โมดูลย่อย หรือ pipeline ในระหว่างพัฒนาเป็นเพียงการยืนยันความพร้อมของต้นแบบ (รวมถึงขั้นตอนในหัวข้อ~\ref{sec:ch3_protocol_e2e}) โดยไม่ขยายจำนวนชุดประเมินเชิงวิจัยเกินกว่าหัวข้อที่กำหนดในส่วนนี้
ตัวเลขผลลัพธ์ของหัวข้อในส่วนนี้นำเสนอในบทที่~\ref{chapter4}

\subsection{วัตถุประสงค์การประเมิน}

วัตถุประสงค์การประเมินสอดคล้องกับวัตถุประสงค์โครงงานในบทที่~\ref{chapter1} โดยมุ่งวัดประสบการณ์ผู้ดูแลต่อชั้นแอปพลิเคชันและคุณภาพ/ประสิทธิภาพของโมดูล LLM ผ่าน MCP ภายใต้ข้อกำหนด Profile-A (หัวข้อ~\ref{sec:ch3_eval_profile})

\subsection{กลุ่มตัวอย่าง การควบคุมสภาพ และหลักการวิเคราะห์เชิงสถิติ}

\textbf{กลุ่มตัวอย่าง:} ผู้เข้าร่วมประเมิน UX เป็นผู้ดูแลหรือผู้มีบทบาทที่กำหนดในพื้นที่ทดสอบ จำนวนและเกณฑ์คัดเลือกรายงานในบทที่~\ref{chapter4}
ผู้ให้คะแนน IRR ต้องได้รับคำอธิบายชุดเครื่องมือและขอบเขตบทบาทก่อนให้คะแนน จำนวนผู้ให้คะแนนอย่างน้อยสองคนเพื่อคำนวณ Cohen's \(\kappa\) หรือใช้ Fleiss' \(\kappa\) เมื่อมีมากกว่าสองคน

\textbf{การควบคุมสภาพ:} ทุกชุดประเมินเชื่อมกับ Profile-A (หัวข้อ~\ref{sec:ch3_eval_profile}) และบันทึกแฮชคอมมิต/แท็กของที่เก็บโค้ด ค่าตัวแปรสภาพแวดล้อมของเซิร์ฟเวอร์ และเวอร์ชันเฟิร์มแวร์อุปกรณ์ในเอกสารประกอบหรือภาคผนวก เพื่อให้การทำซ้ำทางวิศวกรรมเป็นไปได้

\textbf{การวิเคราะห์:} คะแนน Likert รายงานเป็นค่ากลางและการกระจาย (เช่น ส่วนเบี่ยงเวียนมาตรฐานหรือช่วงสี่เหลี่ยม) โดยไม่อ้างการทดสอบสมมติฐานเกี่ยวกับประชากรกว้างหากขนาดตัวอย่างไม่รองรับ
ค่า \(\kappa\) ตีความตามแนวทางทั่วไปของวรรณกรรม (เช่น ค่าต่ำ/กลาง/สูง) และนำเสนอคู่กับคำอธิบายข้อจำกัดของกรณีศึกษา
สำหรับเวลาแฝงของ LLM/MCP รายงานอย่างน้อยค่ามัธยฐาน (p50) และเปอร์เซ็นไทล์ที่ 95 (p95) จากชุดการวัดที่ซ้ำภายใต้สภาพเดียวกัน

\subsection{สรุปสี่ประเด็นประเมิน}
\label{sec:ch3_eval_scope_four}

\begin{enumerate}
    \item \textbf{UX/UI (แบบสอบถาม)} — รวบรวมความคิดเห็นเชิงประสบการณ์ต่อการใช้งานแดชบอร์ด/อินเทอร์เฟซผู้ดูแล รายละเอียดอยู่ในหัวข้อ~\ref{sec:ch3_test_ux}
    \item \textbf{IRR (การเลือกเครื่องมือ MCP)} — วัดความสอดคล้องระหว่างผู้ให้คะแนนว่าในแต่ละบริบทควรเรียกเครื่องมือใด รายละเอียดอยู่ในหัวข้อ~\ref{sec:ch3_eval_irr}
    \item \textbf{ความใกล้เคียงเชิงข้อความ} — วัดความใกล้เคียงระหว่างเอาต์พุต LLM กับข้อความอ้างอิง รายละเอียดอยู่ในหัวข้อ~\ref{sec:ch3_eval_text_sim}
    \item \textbf{เวลาแฝง LLM/MCP} — วัดช่วงเวลาตั้งแต่คำถามของผู้ใช้จนถึงการเรียกเครื่องมือ MCP เสร็จสมบูรณ์ รายละเอียดอยู่ในหัวข้อ~\ref{sec:ch3_eval_llm_latency}
\end{enumerate}

\subsection{การทดสอบเสริม: ความทนทานของการระบุตำแหน่งต่อ RSSI drift แบบจำลอง (Gaussian noise injection)}
\label{sec:ch3_eval_loc_robustness}

เนื่องจากสัญญาณ RSSI มีความผันผวนตามเวลา สภาพแวดล้อม และการบังสัญญาณในทางปฏิบัติ การประเมินเชิงวิศวกรรมจึงเพิ่มการทดสอบเสริมเพื่อสะท้อนกรณี ``ค่า RSSI เปลี่ยนเล็กน้อยในพื้นที่เดิม'' โดยไม่เปลี่ยนแผนผังพื้นที่หรือป้ายกำกับโซน/ห้อง
แนวคิดนี้เชื่อมกับการอภิปรายเรื่อง RSSI drift/temporal variability ในบทที่~\ref{sec:ch2_rssi_drift}
การทดสอบนี้มีวัตถุประสงค์เพื่อ (1) ตรวจสอบแนวโน้มการเสื่อมถอยของความแม่นยำเมื่อ RSSI เปลี่ยน และ (2) เปรียบเทียบพฤติกรรมของ \textbf{KNN} (ตัวแบบหลักของระบบ) กับ \textbf{XGBoost} ซึ่งเป็นตัวแบบ supervised ที่ใช้กันแพร่หลายในงานจำแนกจาก RSSI \cite{Chen2016XGBoost}

\subsubsection{แนวคิดและการสร้างชุดทดสอบที่มีสัญญาณรบกวน}
\label{sec:ch3_eval_loc_noise_injection}

ให้เวกเตอร์ RSSI ของแต่ละตัวอย่างเป็น \(\mathbf{r}\in\mathbb{R}^{d}\) (หลังจัดมิติคงที่และจัดการ missing-RSSI ตามหัวข้อ~\ref{sec:ch3_feature_vector}) แล้วสร้างชุดทดสอบแบบจำลองสัญญาณรบกวนโดย
\[
    \tilde{\mathbf{r}} = \mathbf{r} + \boldsymbol{\epsilon},\qquad \boldsymbol{\epsilon}\sim\mathcal{N}(0,\sigma^2\mathbf{I})
\]
โดยกำหนด \(\sigma\) เป็นระดับการเปลี่ยนแปลงของ RSSI (หน่วย dBm) ในช่วง \(\sigma\in\{2,4,6,8,10,12\}\) เพื่อครอบคลุมตั้งแต่ ``เปลี่ยนเล็กน้อย'' ไปจนถึงกรณีรบกวนสูง ทั้งนี้การสุ่มทำในทุกมิติของ RSSI ต่อหนึ่งตัวอย่าง และทำซ้ำได้ด้วยการกำหนดค่า seed ของกระบวนการสุ่ม
แนวคิดนี้สอดคล้องกับงานศึกษาความไวของระบบ fingerprinting ต่อความแปรผันของ RSSI \cite{Kaemarungsi2012RSSIAnalysis}

\subsubsection{โปรโตคอลการประเมินและตัวชี้วัด}
\label{sec:ch3_eval_loc_protocol}

\begin{enumerate}
    \item \textbf{ชุดฝึก (baseline)} ใช้ radio map/fingerprint ที่เก็บตามโปรโตคอลในหัวข้อ~\ref{sec:ch3_protocol_fingerprint} และฝึกโมเดล KNN ตาม Profile-A (ตารางที่~\ref{tab:ch3_profileA_loc})
    \item \textbf{ตัวแบบเปรียบเทียบ} ฝึก XGBoost บนชุด fingerprint เดียวกัน โดยใช้ฟีเจอร์เวกเตอร์ RSSI แบบเดียวกันเพื่อให้การเปรียบเทียบยุติธรรม (ไม่เพิ่มฟีเจอร์เชิงอนุพันธ์หรือข้อมูลจากภาพ)
    \item \textbf{ชุดทดสอบ (robustness)} สร้าง \(\tilde{\mathbf{r}}\) ด้วย noise injection ตามระดับ \(\sigma\) แล้วประเมินผลทำนายโซน/ห้องเทียบกับป้ายกำกับเดิม (ground truth)
    \item \textbf{ตัวชี้วัด} รายงานอย่างน้อย \textbf{accuracy} ต่อระดับ \(\sigma\) และแสดง \textbf{confusion matrix} ของระดับที่เป็นตัวแทน (เช่น \(\sigma=4\) สำหรับ drift เล็กน้อย) เพื่ออธิบายรูปแบบความผิดพลาด (เช่น สลับโซนที่ติดกัน)
\end{enumerate}

\label{sec:ch3_test_ux}

แผนการประเมินผู้ดูแลหรือผู้เกี่ยวข้องกำหนดเป็นงานที่ต้องทำจริงบนระบบ เช่น การค้นหาข้อมูลย้อนหลัง การตอบสนองต่อการแจ้งเตือน และการใช้งานหน้าจอหลักของแดชบอร์ด
เครื่องมือเก็บข้อมูลอาจใช้แบบสอบถามหรือการสังเกตการณ์ตามที่ออกแบบในโครงงาน โดยไม่ขยายผลเกินขนาดตัวอย่าง

\subsubsection{แบบสอบถามอิเล็กทรอนิกส์ (เช่น Google Form)}

ใช้แบบสอบถามอิเล็กทรอนิกส์ (เช่น Google Form) โครงสร้างประกอบด้วยส่วนข้อมูลทั่วไปของผู้ตอบ (ตามที่เหมาะสมและไม่ละเมิดความเป็นส่วนตัว) รายการงานที่ให้ทดลองบนระบบจริง (เช่น ติดตามโซน รับทราบการแจ้งเตือน เปิดประวัติ) และข้อความแบบมาตราส่วน Likert สำหรับมิติเช่น ความง่ายในการใช้งาน ความชัดเจนของข้อมูล ความเหมาะสมของการแจ้งเตือน และความน่าเชื่อถือโดยรวม
ไม่รายงานค่าตัวเลขหรือสถิติจากแบบสอบถามในบทนี้; ผลสรุปนำเสนอในบทที่~\ref{chapter4}

\subsection{การประเมินโมดูล LLM ผ่าน MCP}
\label{sec:ch3_eval_llm_metrics}

\subsubsection{ความสอดคล้องระหว่างผู้ให้คะแนน (IRR) สำหรับการเลือกเครื่องมือ MCP}
\label{sec:ch3_eval_irr}

กำหนดแผนผังการเข้ารหัสว่าในแต่ละ log การสนทนา ผู้ให้คะแนนควรเลือกเครื่องมือ MCP ที่เหมาะสมจากรายการที่อนุญาตหรือระบุว่าไม่ควรเรียกเครื่องมือ
ระบุจำนวนผู้ให้คะแนน เกณฑ์การคำนวณความสอดคล้อง (เช่น Cohen's \(\kappa\) สำหรับสองผู้ให้คะแนน หรือ Fleiss' \(\kappa\) หากมีมากกว่าสองคน) และกฎการตัดสินเมื่อความเห็นไม่ตรงกัน (adjudication)
ค่าความสอดคล้องและการอภิปรายนำเสนอในบทที่~\ref{chapter4}

\subsubsection{ความใกล้เคียงเชิงข้อความของเอาต์พุต LLM}
\label{sec:ch3_eval_text_sim}

กำหนดข้อความอ้างอิง (reference) สำหรับสถานการณ์หรือคำถามที่เลือกไว้ล่วงหน้า แล้ววัดความใกล้เคียงระหว่างเอาต์พุตของโมเดลกับข้อความอ้างอิงด้วยตัวชี้วัดที่เลือก เช่น ความคล้ายคลึงคอไซน์บนเวกเตอร์ embedding หรือค่า BLEU ตามความเหมาะสมของงาน
ระบุว่าตัวชี้วัดใช้เมื่อใด (เช่น หลังแต่ละคำถามในชุดทดสอบที่ควบคุม) โดยไม่รายงานค่าที่วัดได้ในบทนี้

\subsubsection{เวลาหน่วงแบบปลายทางปลายทางของ LLM}
\label{sec:ch3_eval_llm_latency}

นิยามช่วงเวลาที่วัดจาก \textbf{คำถามของผู้ใช้} จนถึง \textbf{การเรียกเครื่องมือ MCP เสร็จสมบูรณ์} (ได้ผลลัพธ์จากเครื่องมือและส่งกลับในลำดับการทำงาน) ไม่รวมการแสดงผลบนหน้าจอหากแยก timestamp ได้
บันทึก log อย่างน้อย: เวลาที่รับคำถาม เวลาที่เริ่มเรียกเครื่องมือแต่ละครั้ง เวลาที่เครื่องมือตอบกลับ และเวลาที่ลำดับงานเสร็จสิ้น รวมถึงการทดลองอุ่นเครื่อง (warm-up) และจำนวนครั้งซ้ำต่อสถานการณ์
ตัวเลข latency รายงานในบทที่~\ref{chapter4}

\subsection{เกณฑ์สรุปผลและการบันทึกข้อมูล}

เกณฑ์สรุปผลระบุเป็นนิยามตัวชี้วัดที่วัดได้ภายใต้สี่ประเด็นข้างต้น เช่น ค่ามาตราส่วน Likert รายมิติจากแบบสอบถาม UX ค่า \(\kappa\) หรือตัวชี้วัดความสอดคล้องของ IRR ค่าความใกล้เคียงเชิงข้อความ และสถิติเวลาแฝงของ LLM/MCP
การบันทึกข้อมูลการประเมินต้องเชื่อมโยงกับชุดพารามิเตอร์ในหัวข้อ~\ref{sec:ch3_params} เพื่อให้สามารถทำซ้ำและอภิปรายในบทที่~\ref{chapter4} ได้อย่างสอดคล้องกัน

\section{สรุปขั้นตอนการดำเนินงาน}

บทนี้สรุปลำดับการดำเนินงานตั้งแต่การกำหนดบริบทและข้อจำกัด การแปลงวัตถุประสงค์ในบทที่~\ref{chapter1} เป็นความต้องการระบบ การออกแบบโมดูลอุปกรณ์ การระบุตำแหน่งในอาคาร เซิร์ฟเวอร์และฐานข้อมูล แอปพลิเคชัน และการบูรณาการ Home Assistant กับ MCP
บทนี้ยังสรุปแผนการประเมินผล (สี่ประเด็นด้าน UX/UI และ LLM/MCP และการทดสอบเสริมด้านความทนทานของ localization) พร้อมพารามิเตอร์อ้างอิง Profile-A
ขั้นตอนเหล่านี้เป็นโครงร่างสำหรับการติดตั้งและการรายงานผลในบทที่~\ref{chapter4} โดยไม่ทับซ้อนกับเนื้อหาเชิงทฤษฎีและวรรณกรรมในบทที่~\ref{chapter2}
