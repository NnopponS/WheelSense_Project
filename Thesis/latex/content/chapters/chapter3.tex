\subsection{โครงข่าย}
\label{sec:ch3_sensor_network}

เพื่อลดการใช้พลังงานและจัดการความซับซ้อนของการเชื่อมต่อ ระบบจึงแบ่งการสื่อสารออกเป็นสองช่วงหลัก ช่วงแรกเป็นการสื่อสารระหว่างอุปกรณ์สวมใส่และตัวส่งสัญญาณอ้างอิงภายในอาคารกับ M5StickC Plus2 โดยใช้เครือข่าย BLE เป็นหลัก Polar Verity Sense จะทำการส่งข้อมูลสัญญาณชีพผ่านมาตรฐาน BLE GATT Heart Rate Service ในขณะเดียวกัน M5StickC Plus2 จะทำหน้าที่สแกนหาสัญญาณจาก Beacon ที่อยู่รอบตัวเพื่อเก็บค่าระดับความเข้มสัญญาณ (RSSI) ไว้

ช่วงที่สองเป็นการส่งข้อมูลจากเก้าอี้รถเข็นไปยังเซิร์ฟเวอร์ส่วนกลาง เมื่อ M5StickC Plus2 รวบรวมข้อมูลจากการเคลื่อนไหว สัญญาณชีพ และความเข้มสัญญาณบลูทูธได้แล้ว อุปกรณ์จะส่งข้อมูลทั้งหมดเข้าสู่เครือข่าย Wi-Fi ภายในอาคาร โดยเลือกใช้โปรโตคอล MQTT ซึ่งมีความเบาบางและสามารถรักษาเสถียรภาพได้ดีแม้ในสภาวะที่เครือข่ายไร้สายขาดหายชั่วคราว ข้อมูลจะถูกส่งไปยังหัวข้อ (Topic) ที่ระบุตัวตนของอุปกรณ์อย่างชัดเจนเพื่อป้องกันการปะปนกันของข้อมูลจากหลายอุปกรณ์

\subsection{การประมวลผลข้อมูล}
\label{sec:ch3_sensor_processing}

ข้อมูลดิบที่อ่านได้จากเซ็นเซอร์ IMU จะมีลักษณะเป็นความเร่งและอัตราการหมุนทั้ง 3 แกนที่มีความละเอียดสูงและมีอัตราการสุ่มตัวอย่าง (Sampling Rate) ถึง 100 เฮิรตซ์ ข้อมูลในระดับนี้แม้จะมีความแม่นยำสูงแต่ก็ประกอบด้วยสัญญาณรบกวนจากการสั่นสะเทือนตามธรรมชาติของเก้าอี้รถเข็น กระบวนการประมวลผลจึงเริ่มต้นด้วยการคำนวณหาขนาดความเร่งลัพธ์รวม (Acceleration Magnitude) จากทั้ง 3 แกน 

\begin{equation}
a_{\mathrm{mag}} = \sqrt{a_x^2 + a_y^2 + a_z^2}
\label{eq:ch3_accel_mag}
\end{equation}

จากนั้นนำค่าที่ได้ผ่านตัวกรองสัญญาณรบกวนแบบความถี่ต่ำ (Low-pass Filter) เพื่อลดทอนการกระชากของสัญญาณที่เกิดจากความขรุขระของพื้นผิว

\begin{equation}
\bar{a}_t = \alpha \cdot a_{\mathrm{mag},t} + (1-\alpha) \cdot \bar{a}_{t-1}
\label{eq:ch3_lowpass}
\end{equation}

เมื่อได้สัญญาณการเคลื่อนไหวที่นิ่งพอ ระบบจะใช้เงื่อนไขเกณฑ์ค่าขอบ (Threshold-based) เพื่อตัดสินว่าเก้าอี้รถเข็นอยู่ในสถานะนิ่งหรือกำลังเคลื่อนที่ โดยหากขนาดความเร่งหรืออัตราการเปลี่ยนทิศทางมีค่าสูงกว่าที่ตั้งไว้ ระบบจะระบุว่ามีกิจกรรมเคลื่อนที่เกิดขึ้น สุดท้ายข้อมูลที่ผ่านการสรุปผลนี้ พร้อมด้วยข้อมูลแบตเตอรี่ สัญญาณชีพ และรายการ RSSI ของ Beacon จะถูกจัดรูปแบบให้อยู่ในโครงสร้าง JSON ก่อนส่งไปยังเซิร์ฟเวอร์ในอัตรา 20 เฮิรตซ์

\section{การจำแนกตำแหน่ง}
\label{sec:ch3_localization}

\subsection{การประมวลผลข้อมูล}
\label{sec:ch3_localization_processing}

กระบวนการประมวลผลตำแหน่งใช้วิธีการเปรียบเทียบลายนิ้วมือสัญญาณ (Fingerprinting) ซึ่งแบ่งการทำงานเป็น 2 ขั้นตอนหลัก ขั้นตอนแรกคือการสอบเทียบ (Offline Calibration) โดยการรวบรวมค่าความเข้มสัญญาณบลูทูธ (RSSI) จาก Beacon ทุกตัวในแต่ละจุดอ้างอิงบนพื้นที่จริง เพื่อสร้างเป็นแผนที่สัญญาณ (Radio Map) เก็บไว้ในฐานข้อมูล

ขั้นตอนที่สองคือการทำนายตำแหน่งจริง (Online Inference) เมื่อเซ็นเซอร์บนเก้าอี้รถเข็นส่งชุดข้อมูล RSSI ของ Beacon ที่สแกนพบในขณะนั้น ระบบเซิร์ฟเวอร์จะนำชุดข้อมูลดังกล่าวมาคำนวณระยะห่างทางสถิติ (Euclidean Distance) เทียบกับฐานข้อมูลแผนที่สัญญาณที่มีอยู่

\begin{equation}
d_j = \sqrt{\sum_{i=1}^{m} (x_i - r_{ji})^2}
\label{eq:ch3_euclidean}
\end{equation}

จากนั้นระบบจะใช้อัลกอริทึม K-Nearest Neighbors (K-NN) โดยเลือกจุดอ้างอิงที่มีระยะห่างทางสถิติน้อยที่สุดจำนวน $k$ จุด แล้วทำการโหวตน้ำหนักเพื่อหาพิกัดหรือโซนที่น่าจะเป็นไปได้มากที่สุด

\begin{equation}
\hat{y} = \arg\max_{y \in \mathcal{Y}} \sum_{i=1}^{k} \frac{1}{d_i} \cdot \mathbb{1}_{[y_i = y]}
\label{eq:ch3_weighted_knn}
\end{equation}

ในกรณีที่มีสัญญาณขาดหายชั่วคราว ระบบได้กำหนดมาตรการรองรับดังตารางที่~\ref{tab:ch3_fallback}

\section{กลยุทธ์การทดสอบระบบ}
\label{sec:ch3_testing}

เพื่อให้มั่นใจว่าระบบสามารถทำงานได้อย่างถูกต้องและมีเสถียรภาพ กลยุทธ์การทดสอบจึงถูกแบ่งออกเป็น 3 ระดับอย่างเป็นระบบ ระดับแรกคือการทดสอบส่วนหลังบ้าน (Backend Tests) ด้วยเฟรมเวิร์ก Pytest ภายใต้สภาพแวดล้อมจำลองแบบ Docker Compose ซึ่งครอบคลุมทั้ง Unit Tests สำหรับตรวจสอบตรรกะทางธุรกิจและ API รวมถึง Integration Tests สำหรับทดสอบการรับข้อมูล MQTT และการทำงานของเซิร์ฟเวอร์ MCP

ระดับที่สองคือการทดสอบแบบ End-to-End (E2E Tests) ผ่านเครื่องมือ Playwright เพื่อจำลองพฤติกรรมการใช้งานจริงของบุคลากรบนหน้าจอแดชบอร์ด ตั้งแต่กระบวนการเข้าสู่ระบบ การติดตามผู้พักอาศัย การรับทราบการแจ้งเตือน ไปจนถึงการทดสอบการสนทนากับ AI และกลไก Propose-Confirm-Execute

ระดับสุดท้ายคือการบูรณาการการทดสอบอย่างต่อเนื่อง (CI/CD) ผ่าน GitHub Actions ซึ่งจะบังคับให้มีการรัน Backend Tests ทุกครั้งที่มีการแก้ไขโค้ด และรัน E2E Tests เป็นประจำทุกวัน ควบคู่กับการประเมินประสิทธิภาพในมิติต่างๆ เช่น ความต่อเนื่องของข้อมูลเซ็นเซอร์ ความแม่นยำในการระบุตำแหน่ง เสถียรภาพของเซิร์ฟเวอร์ ความถูกต้องของเครื่องมือ AI และประสบการณ์ผู้ใช้งาน (UX) โดยรวม เพื่อรักษาคุณภาพของซอฟต์แวร์ในทุกขั้นตอนการพัฒนา
