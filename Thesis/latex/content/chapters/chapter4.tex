\section{การติดตั้งอุปกรณ์และระบบ}

การดำเนินงานภาคสนามอิงสถาปัตยกรรมและขอบเขตอุปกรณ์ที่สรุปในบทที่~\ref{chapter1} และรายละเอียดวิธีการในบทที่~\ref{chapter3} โดยเริ่มจากการติดตั้งอุปกรณ์หลักในสภาพแวดล้อมจริง ประกอบด้วยโมดูลตรวจจับบนเก้าอี้รถเข็น โหนดภายในอาคาร จุดรวมข้อมูลเครือข่าย (ถ้ามี) และเซิร์ฟเวอร์กลาง เพื่อสร้างเส้นทางข้อมูลตั้งแต่ชั้นเซ็นเซอร์จนถึงแอปพลิเคชันผู้ดูแล
ตัวเลขและพารามิเตอร์การตั้งค่าที่อ้างถึงในบทนี้สอดคล้องกับชุด \textbf{Profile-A} ในหัวข้อ~\ref{sec:ch3_eval_profile} และตารางที่~\ref{tab:ch3_profileA_loc}--\ref{tab:ch3_profileA_runtime} ของบทที่~\ref{chapter3}

\subsection{การติดตั้งอุปกรณ์บนเก้าอี้รถเข็น}

อุปกรณ์ M5StickC Plus2 ถูกติดตั้งกับโครง/ล้อของเก้าอี้รถเข็น พร้อมตั้งค่าเฟิร์มแวร์และพารามิเตอร์สำคัญ เช่น ตัวระบุเก้าอี้รถเข็น (Wheel ID) ค่าปรับเทียบเซ็นเซอร์ IMU และช่วงเวลาการส่งข้อมูล telemetry ก่อนทดสอบการอ่านเซ็นเซอร์และการส่งข้อมูลเบื้องต้น ส่วน Polar Verity Sense ถูกจัดวางตามแผนการทดลองเพื่อเก็บสัญญาณทางสรีรศาสตร์ควบคู่กับข้อมูลการเคลื่อนไหว

\subsection{การติดตั้งโหนดและจุดรวมข้อมูลภายในอาคาร}

โหนด ESP32-S3 พร้อมกล้องถูกติดตั้งตามจุดใช้งานสำคัญเพื่อครอบคลุมมุมมองภาพ และกำหนดค่าที่เกี่ยวข้องกับการส่งภาพและการสื่อสารเครือข่ายไปยัง MQTT Broker ตามการออกแบบ pipeline ในบทที่~\ref{chapter3} โดยมีการทดสอบความต่อเนื่องของการเชื่อมต่อและความถูกต้องของข้อมูลที่ส่งผ่านแต่ละชั้น

\subsection{การติดตั้งเซิร์ฟเวอร์}

Raspberry Pi 5 (RAM 8GB) ถูกใช้เป็นแพลตฟอร์มเซิร์ฟเวอร์กลาง โดยติดตั้งระบบปฏิบัติการ เครื่องมือคอนเทนเนอร์ และบริการหลักของระบบ เช่น MQTT Broker ตัวรับข้อมูล (collector) ฐานข้อมูล API และแอปพลิเคชันเว็บ จากนั้นทดสอบการทำงานของบริการร่วมกันในสภาพแวดล้อมเดียว

\section{การทดสอบระบบ}

\subsection{ผลการทดสอบฮาร์ดแวร์}

การทดสอบฮาร์ดแวร์ครอบคลุมความสามารถในการอ่านข้อมูลเซ็นเซอร์และ RSSI จากอุปกรณ์บนเก้าอี้ รวมถึงการส่งภาพจากโหนด ESP32-S3 ผ่านเครือข่ายไปยัง MQTT Broker ผลโดยรวมพบว่าอุปกรณ์แต่ละส่วนทำงานได้ แต่เมื่อขยายระบบในระดับหลายโหนดหรือหลายอุปกรณ์พร้อมกัน ความเสถียรของการเชื่อมต่อไร้สายและเวลา recovery หลังขาดสัญญาณยังเป็นข้อจำกัดที่ต้องติดตามต่อเนื่อง

\subsection{ผลการทดสอบซอฟต์แวร์}

ฝั่งซอฟต์แวร์มีการทดสอบเฟิร์มแวร์ การสื่อสาร MQTT การจัดการฐานข้อมูล REST API และแอปพลิเคชันเว็บแบบเรียลไทม์ ผลการทดสอบแสดงว่าระบบแสดงผลและจัดเก็บข้อมูลได้ตามฟังก์ชันหลัก แต่ประสิทธิภาพโดยรวมยังได้รับผลจากความหน่วงในชั้นเครือข่ายและชั้นบริการเมื่อมีปริมาณข้อมูลสูงหรือเกิดการแย่งชิงทรัพยากรบนเซิร์ฟเวอร์

\subsection{ผลการทดสอบแบบบูรณาการ}

การทดสอบ End-to-End ตั้งแต่อุปกรณ์บนเก้าอี้รถเข็นไปจนถึงแอปพลิเคชันผู้ดูแลแสดงให้เห็นประเด็นสำคัญ 4 ด้าน:
\begin{enumerate}
    \item Latency ของระบบโดยรวมสูงกว่าที่ต้องการสำหรับงานเรียลไทม์เข้มข้น
    \item Throughput ลดลงเมื่อเพิ่มจำนวนเก้าอี้รถเข็นพร้อมกัน
    \item ความถูกต้องของข้อมูลลดลงในบางช่วงเมื่อเกิดความล่าช้าหรือ packet loss
    \item เวลา recovery หลังการหลุดการเชื่อมต่อยังยาวและกระทบความต่อเนื่องของบริการ
\end{enumerate}

\section{การทดสอบในสภาพแวดล้อมจริง}

มีการประเมินการทำงานจริงในหลายพื้นที่ของอาคาร ทั้งการเคลื่อนที่ภายในห้องและการเคลื่อนที่ข้ามห้อง เพื่อทดสอบความแม่นยำการระบุตำแหน่ง ความเสถียรระยะเวลาทำงานต่อเนื่อง และพฤติกรรมระบบในสภาวะที่มีสิ่งกีดขวางหรือสัญญาณรบกวน

\subsection{การตีความพฤติกรรมด้วย AI และการประเมินโดยผู้เชี่ยวชาญ}

ในระยะนี้มีการวางแนวทางใช้ข้อมูลผสานจาก IMU, การเคลื่อนที่ และตำแหน่งในอาคารเพื่อจำแนกพฤติกรรม เช่น การเคลื่อนที่ปกติ การหยุดนิ่งผิดปกติ หรือการเคลื่อนที่ในพื้นที่เสี่ยง พร้อมกำหนดกระบวนการประเมินผลโดยผู้เชี่ยวชาญ (human-in-the-loop) เพื่อตรวจสอบความถูกต้องของผลการตีความและใช้ข้อเสนอแนะในการปรับปรุงโมเดลต่อไป

\section{อภิปรายผล}

ผลการดำเนินงานยืนยันว่าระบบต้นแบบสามารถเชื่อมโยงอุปกรณ์ เครือข่าย และซอฟต์แวร์เป็นสายงานเดียวได้จริง และรองรับการติดตาม/แสดงผลข้อมูลได้ตามเป้าหมายเชิงต้นแบบ อย่างไรก็ตาม ข้อจำกัดด้าน latency, scalability และการกู้คืนหลังขาดการเชื่อมต่อ ชี้ให้เห็นความจำเป็นของการปรับสถาปัตยกรรมและการจัดการคิวข้อมูลในระยะถัดไป โดยเฉพาะเมื่อเพิ่มจำนวนอุปกรณ์และบริการที่ทำงานบน Raspberry Pi 5 พร้อมกัน
