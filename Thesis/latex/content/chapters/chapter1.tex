\section{ที่มา และความสำคัญ}
\label{sec:ch1_significance}

ผู้ใช้เก้าอี้รถเข็นในพื้นที่ภายในอาคารยังเผชิญข้อจำกัดด้านความปลอดภัย ความสะดวก และความต่อเนื่องในการเข้าถึงบริการ โดยเฉพาะเมื่อต้องการข้อมูลตำแหน่งและสถานะการเคลื่อนที่แบบเรียลไทม์ เช่น ในบ้าน โรงพยาบาล และสถานดูแล ซึ่งสอดคล้องกับบริบทความท้าทายด้านสุขภาวะของคนพิการในภาพรวม \cite{WHO_Disability_Health}
ด้วยเหตุนี้ แนวคิดสภาพแวดล้อมอัจฉริยะและการนำ IoT มาช่วยสนับสนุนการดำรงชีวิตประจำวันจึงมีความสำคัญ อย่างไรก็ตาม ในเชิงปฏิบัติการ ระบบที่พบได้ทั่วไปยังไม่ครบถ้วนเมื่อต้องเชื่อม ``การรับรู้เชิงกายภาพ'' ของผู้ใช้เข้ากับ ``บริบทของสถานที่'' และ ``การทำงานอัตโนมัติ'' ให้ทำงานสอดคล้องกัน \cite{Vimarlund2021AAL_challenges}

ในเชิงเทคนิค การติดตามตำแหน่งด้วย GPS มีข้อจำกัดเมื่อใช้งานในอาคาร \cite{Indoor_GPS_Limitations} ส่วนแนวทางระบุตำแหน่งที่มีความแม่นยำสูงบางประเภทมีต้นทุนและความซับซ้อนในการติดตั้งสูง จึงไม่เหมาะกับการพัฒนาระบบต้นแบบที่ต้องการความยืดหยุ่นและการขยายผลได้จริง
โครงงานนี้จึงเลือกแนวทางที่เน้นสมดุลระหว่างประสิทธิภาพและความเป็นไปได้ในการใช้งาน โดยใช้ BLE beacon และ RSSI fingerprinting ร่วมกับอัลกอริทึม KNN สำหรับ indoor localization \cite{BLE_Fingerprint_Survey_IEEE,WiFi_BLE_IPS_Systematic_Review} อุปกรณ์บนเก้าอี้รถเข็นรวบรวมข้อมูล BLE/RSSI และส่งขึ้นเซิร์ฟเวอร์ร่วมกับ telemetry ส่วนโหนดกล้อง \texttt{Node\_Tsimcam} (ESP32-S3) ส่ง \textbf{ภาพนิ่ง (photo)} เพื่อช่วยให้ผู้ดูแลรับรู้บริบท โดยใช้การส่งแบบแบ่งชิ้นส่วน (chunked payload) ผ่าน MQTT และมีแนวทางสำรองด้วยการอัปโหลดผ่าน HTTP เมื่อจำเป็น ดังนั้นสถาปัตยกรรมจึงแยกบทบาทการส่งข้อมูลระหว่างอุปกรณ์บนเก้าอี้ โหนดกล้อง และการประมวลผลบนเซิร์ฟเวอร์อย่างชัดเจน

อย่างไรก็ตาม การพัฒนาระบบสนับสนุนผู้ใช้เก้าอี้รถเข็นให้ใช้งานได้จริงยังมีข้อจำกัดที่พบซ้ำในทางปฏิบัติ ดังนี้
\begin{enumerate}
    \item ระบบอัตโนมัติในบ้าน (Home Automation) ส่วนใหญ่เน้นการควบคุมอุปกรณ์ไฟฟ้า แต่ยังไม่ผสาน indoor localization และข้อมูลพฤติกรรมของผู้ใช้แบบเรียลไทม์อย่างเป็นระบบ
    \item การติดตามตำแหน่งด้วย GPS มีข้อจำกัดด้านความแม่นยำเมื่อใช้ภายในอาคาร เนื่องจากสัญญาณถูกรบกวนจากโครงสร้างอาคาร \cite{Indoor_GPS_Limitations} จึงไม่เหมาะกับการระบุตำแหน่งแบบต่อเนื่องในพื้นที่ในร่ม
    \item งาน indoor localization จำนวนมากมุ่งเน้นความแม่นยำของตำแหน่ง \cite{BLE_Fingerprint_Survey_IEEE,WiFi_BLE_IPS_Systematic_Review} แต่ไม่ได้ออกแบบให้เชื่อมต่อกับระบบดูแลผู้ใช้เก้าอี้รถเข็นแบบครบสายงาน (end-to-end) นอกจากนี้ แนวทางที่ผสาน sensing, localization, application, automation และการช่วยตัดสินใจเชิงอัตโนมัติบนแพลตฟอร์มต้นทุนเหมาะสมและติดตั้งได้จริงยังมีจำกัด
\end{enumerate}

เมื่อพิจารณาเชิงการนำไปใช้งานจริง งานวิจัยบางส่วนยังต้องพึ่งโครงสร้างพื้นฐานเฉพาะทางหรือมีขั้นตอน calibration ที่ซับซ้อน โครงงานนี้จึงกำหนดแนวทางที่ใช้ฮาร์ดแวร์และซอฟต์แวร์ที่เข้าถึงได้ง่าย และเน้นความสามารถในการทำซ้ำ (reproducibility) รวมถึงการดูแลรักษาในระยะยาว
รายละเอียดการทบทวนวรรณกรรม การเปรียบเทียบกับงานที่เกี่ยวข้อง และการอธิบายช่องว่างเชิงลึก นำเสนอในบทที่~\ref{chapter2}

จากข้อจำกัดดังกล่าว โครงงานกำหนดให้อุปกรณ์บนเก้าอี้ (M5StickC Plus2 ร่วมกับ IMU และ Polar Verity Sense) ทำหน้าที่หลักด้าน telemetry การตรวจจับเหตุการณ์เชิงการเคลื่อนที่ และการรวบรวมข้อมูล BLE/RSSI เพื่อส่งขึ้นเซิร์ฟเวอร์สำหรับ localization ส่วนโหนดกล้อง \texttt{Node\_Tsimcam} (ESP32-S3) ทำหน้าที่เก็บและส่ง \textbf{ภาพนิ่ง} ขึ้นเซิร์ฟเวอร์เพื่อช่วยยืนยันบริบทให้ผู้ดูแล
การแยกบทบาทดังกล่าวช่วยลดความคลุมเคือในการวิเคราะห์ปัญหาเชิงระบบ ลด coupling ระหว่างโมดูล และเอื้อต่อการขยายผลในอนาคต

ประเด็นเชิงระบบที่โครงงานนี้มุ่งจัดการในสภาพแวดล้อมจริง ได้แก่ (1) การระบุตำแหน่งระดับโซน/ห้องในอาคาร (2) การตรวจจับเหตุการณ์ผิดปกติจากสัญญาณการเคลื่อนที่/สรีรศาสตร์ (3) การส่งต่อผลลัพธ์ไปสู่ผู้ดูแลและระบบอัตโนมัติให้ทันเวลา

กล่าวโดยสรุป ระบบต้องรองรับทั้งการเก็บข้อมูล การระบุตำแหน่ง ชั้นแอปพลิเคชัน และกลไกช่วยตัดสินใจ/ทำงานอัตโนมัติ ภายใต้ข้อจำกัดด้านต้นทุนและความซับซ้อนของการติดตั้ง ความท้าทายหลักอยู่ที่การประสานข้อมูลหลายแหล่งให้ตีความร่วมกันได้ในระดับเวลาใกล้เคียงกัน เพราะข้อมูล telemetry และ localization จากอุปกรณ์บนเก้าอี้กับภาพบริบทจากโหนดกล้อง \texttt{Node\_Tsimcam} มีลักษณะและความถี่การอัปเดตต่างกัน หากขาดการออกแบบ pipeline ที่เหมาะสม ระบบจะตอบสนองเชิงบริบทได้ไม่ทันหรือเกิดการแจ้งเตือนที่คลาดเคลื่อน

งานนี้มีความสำคัญทั้งในบริบทไทยที่ต้องการระบบติดตั้งและดูแลรักษาได้จริง และในเชิงวิศวกรรมที่ต้องบูรณาการหลายชั้นเทคโนโลยีให้ทำงานร่วมกันอย่างมีเสถียรภาพ
การบรรลุผลในระดับต้นแบบจึงไม่ใช่เพียงการพัฒนาอัลกอริทึมเดี่ยว แต่เป็นการจัดการ trade-off ระหว่างความแม่นยำ ความหน่วง ความง่ายในการติดตั้ง และต้นทุนรวมของระบบ
ในทางปฏิบัติการ ผู้ดูแลควรได้รับข้อมูลเชิงปฏิบัติการจากตำแหน่งล่าสุด รูปแบบการเคลื่อนที่ และสถานะผิดปกติ ผ่านช่องทางเดียว และเชื่อมต่อสู่ Home Assistant เพื่อกำหนด automation ตามบริบทได้

โครงงานพัฒนาต้นแบบที่บูรณาการองค์ประกอบดังกล่าวให้ทำงานร่วมกันแบบ end-to-end โดยให้ความสำคัญกับความต่อเนื่องของ data pipeline และการสื่อสารผลลัพธ์ที่ผู้ดูแลนำไปใช้ตัดสินใจได้ทันที รายละเอียดวัตถุประสงค์ ขอบเขตงานวิจัย แผนการดำเนินงาน ระยะเวลาดำเนินงาน และประโยชน์ที่คาดว่าจะได้รับ จัดทำในหัวข้อถัดไป

\phantomsection\label{sec:ch1_contributions}
เพื่อให้การประเมินในบทที่~\ref{chapter4} มีความหมายต่อการนำไปใช้งานจริง โครงงานนี้กำหนด ``สิ่งใหม่'' (contributions) ในระดับระบบ ไม่ใช่เพียงการเลือกอัลกอริทึมเดี่ยว โดยสรุปเป็นประเด็นดังนี้
\begin{enumerate}
    \item \textbf{สถาปัตยกรรมต้นแบบแบบ end-to-end ที่ตรวจสอบย้อนกลับได้} — ออกแบบให้ข้อมูลจากอุปกรณ์บนเก้าอี้ (IMU, RSSI, battery, สัญญาณสรีระศาสตร์) ไหลผ่าน MQTT$\rightarrow$บริการรับเข้า$\rightarrow$ฐานข้อมูล$\rightarrow$API$\rightarrow$แดชบอร์ดผู้ดูแล และสามารถอธิบายความหน่วง/ความสดใหม่จาก log ได้ (รายละเอียดในบทที่~\ref{chapter3})
    \item \textbf{การระบุตำแหน่งแบบ RSSI fingerprinting + KNN พร้อม baseline และแนวทางความทนทาน} — ใช้ KNN เป็นตัวแบบหลักในบริบทที่ต้องการสมดุลระหว่างต้นทุนและความยืดหยุ่น พร้อม baseline เชิงปฏิบัติการ (max-RSSI) และการทดสอบความทนทานต่อ RSSI drift แบบจำลอง โดยเปรียบเทียบกับตัวแบบ supervised อย่าง XGBoost เพื่ออภิปรายข้อจำกัดอย่างมีหลักฐาน
    \item \textbf{การเสริมบริบทด้วยภาพนิ่งที่แยกบทบาทจาก localization} — ใช้โหนดกล้อง \texttt{Node\_Tsimcam} ส่งภาพนิ่งเพื่อช่วยยืนยันบริบทต่อผู้ดูแล โดยไม่ใช้เป็นฟีเจอร์หลักของโมเดลตำแหน่ง เพื่อลดความเสี่ยงด้านความเป็นส่วนตัวเมื่อเทียบกับการเฝ้าระวังด้วยวิดีโอต่อเนื่อง
    \item \textbf{การเชื่อมระบบอัตโนมัติและการช่วยตีความภายใต้กรอบความปลอดภัย} — ผสาน Home Assistant สำหรับ context-aware automation และเชื่อม local LLM ผ่าน MCP ภายใต้รูปแบบ propose--confirm--execute เพื่อให้การกระทำที่เปลี่ยนสถานะต้องมีการยืนยันก่อน ลดความเสี่ยงจากการทำงานอัตโนมัติที่ผิดพลาด
    \item \textbf{ช่องทางใช้งานหลายรูปแบบที่สอดคล้องกัน} — มี web dashboard เป็นช่องทางหลักของผู้ดูแล และมีแอปมือถือเป็นช่องทางเสริมสำหรับการใช้งานพกพาและโหมด \textit{walking} โดยใช้สัญญาข้อมูล (topics/payload) และการยืนยันตัวตนร่วมกับระบบเดียวกัน
\end{enumerate}

\section{วัตถุประสงค์}
\label{sec:ch1_objectives}

\begin{enumerate}
    \item พัฒนาอุปกรณ์บนเก้าอี้รถเข็นด้วย M5StickC Plus2 ร่วมกับ IMU เพื่อเก็บข้อมูลการเคลื่อนที่ และใช้ Polar Verity Sense จำนวน 2 ชุด เพื่อเก็บข้อมูลทางสรีรศาสตร์
    \item พัฒนากลไกส่งข้อมูลจากอุปกรณ์ไปยังเซิร์ฟเวอร์ส่วนกลางสำหรับการประมวลผลและแสดงผลแบบต่อเนื่อง
    \item พัฒนาระบบระบุตำแหน่งภายในอาคารโดยใช้ BLE beacon, RSSI fingerprinting และ KNN บนเซิร์ฟเวอร์ โดยให้ M5StickC Plus2 รวบรวมข้อมูล BLE/RSSI และส่งขึ้นเซิร์ฟเวอร์ร่วมกับ telemetry
    \item พัฒนา application สำหรับการติดตามแบบเรียลไทม์ การแสดงสถานะตำแหน่ง และการวิเคราะห์ข้อมูล โดยใช้ Next.js, React และ Tailwind CSS
    \item พัฒนาการตรวจจับเหตุการณ์ผิดปกติด้วยตรรกะแบบ rule-based จากสัญญาณ accelerometer
    \item บูรณาการระบบกับ Home Assistant เพื่อรองรับการทำงานอัตโนมัติที่อิงบริบทการใช้งาน
    \item ประยุกต์ใช้ local LLM (Gemma 4B) ผ่าน MCP (Model Context Protocol) เพื่อช่วยตีความข้อมูลและสนับสนุนการแจ้งเตือนเชิงรุก
    \item ทดสอบและประเมินระบบต้นแบบในมิติความถูกต้อง ความเสถียร และความเหมาะสมต่อการนำไปใช้งานจริง
\end{enumerate}

\section{ขอบเขตงานวิจัย}
\label{sec:ch1_scope}

ในด้านองค์ความรู้ ขอบเขตงานวิจัยครอบคลุมหัวข้อดังนี้
\begin{enumerate}
    \item ระบบฝังตัวและการสื่อสารไร้สายสำหรับ IoT
    \item การอ่านและประมวลผลข้อมูล IMU และ accelerometer สำหรับตรวจจับการเคลื่อนที่
    \item การวัดสัญญาณทางสรีรศาสตร์ด้วย Polar Verity Sense (จำนวน 2 ชุด)
    \item Bluetooth Low Energy (BLE) และ BLE beacon สำหรับงานระบุตำแหน่ง
    \item แนวคิด RSSI fingerprinting และอัลกอริทึม KNN สำหรับ indoor localization
    \item สถาปัตยกรรมเซิร์ฟเวอร์บน Raspberry Pi 5 (8GB RAM) และการจัดการบริการด้วยแนวทาง containerized deployment
    \item การพัฒนา web application ด้วย Next.js, React และ Tailwind CSS
    \item โปรโตคอล MQTT และการเชื่อมต่อระบบอัตโนมัติผ่าน Home Assistant
    \item การออกแบบกฎเชิงตรรกะ (rule-based logic) สำหรับ anomaly detection จาก accelerometer
    \item การใช้งาน local LLM (Gemma 4B) ผ่าน MCP (Model Context Protocol) เพื่อสนับสนุนการตีความข้อมูลในระบบ
\end{enumerate}

ขอบเขตของระบบตามโมดูล สรุปได้ดังนี้
\begin{enumerate}
    \item \textbf{ส่วนอุปกรณ์บนเก้าอี้:} ใช้ M5StickC Plus2 และ IMU สำหรับตรวจจับการเคลื่อนที่ พร้อมรับข้อมูลเสริมจาก Polar Verity Sense จำนวน 2 ชุด
    \item \textbf{ส่วนระบุตำแหน่งในอาคาร:} ใช้ BLE beacon และข้อมูล BLE/RSSI ที่ M5StickC Plus2 ส่งขึ้นเซิร์ฟเวอร์ ประมวลผล fingerprinting ด้วย KNN บนเซิร์ฟเวอร์ โดยมีโหนดกล้อง \texttt{Node\_Tsimcam} (ESP32-S3) ส่งภาพนิ่งเพื่อสนับสนุนการรับรู้บริบท (ไม่ได้ใช้เป็นฟีเจอร์หลักของโมเดล KNN)
    \item \textbf{ส่วนเซิร์ฟเวอร์:} ติดตั้งบน Raspberry Pi 5 (8GB RAM) สำหรับรับข้อมูล จัดเก็บ และให้บริการแก่ application และระบบอัตโนมัติ
    \item \textbf{ส่วนแอปพลิเคชัน:} พัฒนา web application ด้วย Next.js, React และ Tailwind CSS สำหรับ real-time monitoring และการวิเคราะห์ข้อมูล
    \item \textbf{ส่วนตรวจจับเหตุการณ์ผิดปกติ:} ใช้ rule-based logic จาก accelerometer (เช่นค่าแกน z) เพื่อรองรับการแจ้งเตือนเบื้องต้น
    \item \textbf{ส่วน AI assistance:} ใช้ local LLM (Gemma 4B) ผ่าน MCP (Model Context Protocol) สำหรับการสรุปและตีความข้อมูล โดยไม่รวม cloud LLM
    \item \textbf{ส่วนระบบอัตโนมัติ:} เชื่อมต่อกับ Home Assistant เพื่อสร้าง automation ตามเหตุการณ์และตำแหน่ง
    \item \textbf{ส่วน mobile application:} พัฒนาแอปพลิเคชันมือถือแบบข้ามแพลตฟอร์ม (React Native + Expo) เป็นส่วนหนึ่งของระบบปัจจุบัน เพื่อรองรับผู้ใช้ที่ไม่ได้นั่งเก้าอี้รถเข็นและกรณีผู้ใช้เก้าอี้รถเข็นที่ต้องการโหมดการใช้งานนอกเก้าอี้ (\textit{walking mode}) โดยแอปสามารถสแกน BLE/RSSI และเชื่อมต่อ Polar Verity Sense แล้วส่ง telemetry ผ่าน MQTT ภายใต้หัวข้อ \texttt{WheelSense/mobile/\{device\_id\}/telemetry} และรับค่าคอนฟิกจาก \texttt{WheelSense/config/...} ทั้งนี้ web application ยังเป็นช่องทางหลักสำหรับการติดตามและการจัดการเชิงปฏิบัติการของผู้ดูแล
    \item \textbf{ส่วนประเมินผล:} ครอบคลุมความถูกต้องของ localization ความน่าเชื่อถือของ pipeline ข้อมูล และประโยชน์เชิงการใช้งานของระบบ
\end{enumerate}

\phantomsection\label{sec:ch1_boundaries}
เพื่อไม่ให้เกิดการตีความเกินขอบเขต (overclaiming) โครงงานนี้กำหนดข้อจำกัดและสิ่งที่ไม่รวมในขอบเขตงานวิจัย ดังนี้
\begin{enumerate}
    \item \textbf{ความละเอียดตำแหน่ง:} มุ่งที่ระดับ \textbf{โซน/ห้อง} ไม่มุ่งพิกัดระดับเซนติเมตรหรือการนำทางละเอียด
    \item \textbf{เชิงคลินิก:} การแจ้งเตือนและการสรุปข้อมูลเป็นการสนับสนุนการตัดสินใจของผู้ดูแล \textbf{ไม่ใช่} การวินิจฉัยหรือคำแนะนำทางการแพทย์
    \item \textbf{ภาพจากกล้อง:} ใช้ภาพนิ่งเพื่อบริบท ไม่พัฒนาโมเดลวิชั่นหรือระบบเฝ้าระวังวิดีโอต่อเนื่อง
    \item \textbf{AI/LLM:} ใช้ local LLM ภายใต้เครื่องมือที่จำกัดและการยืนยันก่อนกระทำ ไม่รวมการพึ่ง cloud LLM และไม่เปิดเครื่องมืออิสระนอกนโยบายสิทธิ์
    \item \textbf{การประเมิน:} ผลเชิงตัวเลขอ้างอิงสภาพแวดล้อมทดสอบและ Profile-A ในบทที่~\ref{chapter3}; การสรุปผลต้องอภิปรายข้อจำกัดด้านสภาพแวดล้อมและการทำซ้ำ
\end{enumerate}

\section{แผนการดำเนินงาน}
\label{sec:ch1_methodology}

\begin{enumerate}
    \item วิเคราะห์ความต้องการเชิงระบบและกำหนด use case หลักของผู้ใช้เก้าอี้รถเข็นและผู้ดูแล
    \item ออกแบบสถาปัตยกรรมต้นแบบโดยแยกบทบาทระหว่าง \mbox{data acquisition, localization, server, application และ automation}
    \item พัฒนาและทดสอบโมดูลอุปกรณ์บนเก้าอี้ (M5StickC Plus2 + IMU + Polar Verity Sense จำนวน 2 ชุด)
    \item ออกแบบและติดตั้งโหนดกล้อง \texttt{Node\_Tsimcam} (ESP32-S3) สำหรับส่งภาพนิ่งขึ้นเซิร์ฟเวอร์ และติดตั้ง BLE beacon พร้อม pipeline ให้ M5StickC Plus2 ส่งข้อมูล RSSI เพื่อสร้าง fingerprint dataset
    \item พัฒนาโมดูล KNN สำหรับ localization และปรับพารามิเตอร์ให้เหมาะสมกับพื้นที่ทดสอบ
    \item พัฒนา server pipeline บน Raspberry Pi 5 เพื่อรับข้อมูล จัดเก็บ และให้บริการข้อมูลแบบเรียลไทม์
    \item พัฒนา web application ด้วย Next.js, React และ Tailwind CSS พร้อมเชื่อมต่อข้อมูลจากเซิร์ฟเวอร์
    \item พัฒนา rule-based anomaly detection จาก accelerometer และเชื่อมระบบแจ้งเตือน
    \item เชื่อมต่อ Home Assistant เพื่อกำหนด automation ตามบริบทการใช้งาน
    \item ผสาน local LLM (Gemma 4B) ผ่าน MCP เพื่อช่วยตีความข้อมูลและสนับสนุนการตัดสินใจ
    \item ทดสอบแบบบูรณาการทั้งระบบ วิเคราะห์ผล ปรับปรุง และสรุปผลเชิงวิศวกรรม
\end{enumerate}

จากแผนการดำเนินงานข้างต้น โครงงานต้องอาศัยทรัพยากรที่ครอบคลุมทั้งฮาร์ดแวร์ ซอฟต์แวร์ พื้นที่และบริบทการทดสอบ และบุคลากร โดยสรุปได้ดังนี้

\FloatBarrier
\phantomsection\label{sec:ch1_research_questions}
เพื่อให้การรายงานผลเชิงตัวเลขในบทที่~\ref{chapter4} มีกรอบตีความที่ชัดเจน โครงงานกำหนดคำถามและสมมติฐานเชิงวิศวกรรมที่ตรวจสอบได้ ดังนี้
\begin{enumerate}
    \item \textbf{RQ1 (pipeline readiness):} ระบบสามารถส่งผ่านข้อมูลตั้งแต่ sensing ถึงการแสดงผลบนแดชบอร์ดได้ต่อเนื่องและตรวจสอบย้อนกลับได้ภายใต้สภาพเครือข่ายทดสอบหรือไม่
    \item \textbf{RQ2 (localization practicality):} การระบุตำแหน่งระดับโซน/ห้องด้วย RSSI fingerprinting + KNN ให้ผลที่เพียงพอต่อการใช้งานเชิงบริบทของผู้ดูแลภายใต้ \textbf{Profile-A} (ชุดพารามิเตอร์อ้างอิงที่ระบุในบทที่~\ref{chapter3}) หรือไม่ เมื่อเทียบกับ baseline max-RSSI
    \item \textbf{RQ3 (robustness):} เมื่อ RSSI เปลี่ยนแปลงเล็กน้อย (drift) วิธี KNN ยังคงรักษาแนวโน้มความแม่นยำได้ในระดับที่ใช้งานได้หรือไม่ และเสื่อมถอยต่างจาก XGBoost อย่างไรภายใต้การจำลองสัญญาณรบกวน
    \item \textbf{RQ4 (caregiver UX):} ผู้ดูแลสามารถทำงานหลัก (ดูตำแหน่งล่าสุด ตอบสนองต่อแจ้งเตือน ดูประวัติย่อ และดูภาพบริบทเมื่อจำเป็น) ได้อย่างมีประสิทธิภาพและเข้าใจข้อมูลเพียงพอหรือไม่
    \item \textbf{RQ5 (AI safety and usefulness):} การใช้ LLM ผ่าน MCP ภายใต้กรอบยืนยันก่อนกระทำช่วยให้สรุป/ตีความข้อมูลได้โดยยังคงความปลอดภัยและความตรวจสอบย้อนหลัง (auditability) เพียงพอหรือไม่
\end{enumerate}

\section{ประโยชน์ที่คาดว่าจะได้รับ}
\label{sec:ch1_benefits}

\begin{enumerate}
    \item ได้ระบบต้นแบบสภาพแวดล้อมอัจฉริยะสำหรับผู้ใช้เก้าอี้รถเข็นที่ทำงานแบบ end-to-end ได้จริง ตั้งแต่ sensing ถึง visualization
    \item ได้ระบบ indoor localization ที่ใช้ RSSI fingerprinting และ KNN บนเซิร์ฟเวอร์จากข้อมูลที่ M5StickC Plus2 ส่งขึ้นมา ซึ่งสามารถนำไปประยุกต์ใช้กับพื้นที่จริงได้
    \item ได้โมดูลตรวจจับเหตุการณ์ผิดปกติแบบ rule-based จาก accelerometer เพื่อเพิ่มความปลอดภัยในการใช้งาน
    \item ได้ web application ที่ช่วยให้ผู้ดูแลติดตามข้อมูลและเหตุการณ์สำคัญได้แบบเรียลไทม์
    \item ได้การเชื่อมต่อ Home Assistant และ local LLM (Gemma 4B) ผ่าน MCP เพื่อสนับสนุน automation และการตีความข้อมูล
    \item ได้แนวทางขยายผลสู่การใช้งานในบ้าน โรงพยาบาล และสถานดูแล พร้อมแอปพลิเคชันมือถือแบบข้ามแพลตฟอร์ม (React Native + Expo) ที่ทำงานร่วมกับระบบเดียวกัน เพื่อรองรับโหมด \textit{walking} และการเก็บข้อมูลจากอุปกรณ์พกพาในบริบทที่ไม่สามารถพึ่งอุปกรณ์บนเก้าอี้ได้ตลอดเวลา
    \item ผู้พัฒนาได้รับองค์ความรู้เชิงบูรณาการด้านระบบฝังตัว localization algorithm ระบบเซิร์ฟเวอร์ และการพัฒนาแอปพลิเคชันสมัยใหม่
\end{enumerate}
